# 코드 리뷰 정책

## 리뷰 원칙

- 모든 PR은 최소 1명의 리뷰어 승인 필요
- 본인 코드 self-review 불가 (다른 사람이 반드시 리뷰)
- 리뷰어는 코드 품질뿐 아니라 **운영 영향**도 확인

## 리뷰 체크포인트

### 기본
- [ ] 요구사항(티켓)과 일치하는가
- [ ] 테스트가 포함되어 있는가
- [ ] 기존 컨벤션을 따르는가

### 운영 영향
- [ ] DB/SP 변경이 있는가 → 별도 승인 (단, 프론트엔드 전용 서비스 PR은 체크 대상에서 제외)
- [ ] 외부 시스템 연동 변경이 있는가
- [ ] 레거시 경계를 넘는 변경이 있는가 → LEGACY_BOUNDARY.md 확인

### 현대화 관련
- [ ] 신규 코드에서 레거시 DB/SP 직접 접근이 없는가 (백엔드/API/배치/DB 레포만 해당)
- [ ] adapter를 통해 접근하는가
- [ ] 서비스 하네스 갱신이 필요한 변경인가

## 서비스별 DB/SP 체크 적용

- 프론트엔드 전용 서비스 PR은 DB/SP 변경 여부와 SP 직접 접근 여부를 코드리뷰 체크포인트에서 제외한다.
- 백엔드/API/배치/DB 스크립트 레포 또는 프론트 PR에 DB 스크립트·SP 호출 계약 변경이 함께 포함된 경우에는 DB/SP 체크를 적용한다.

## DB/SP 변경 추가 리뷰 항목

DB 또는 SP 변경이 포함된 PR은 아래를 반드시 포함:

- 영향 테이블 목록
- 입력/출력 계약 변경 사항

> 롤백 스크립트·최소 재현 데이터셋·smoke test 시나리오는 **프로덕션 배포 시점**에 별도 운영 승인 플로우에서 확인한다. PR 단계에서는 차단 기준으로 삼지 않는다.
