# AI 사용 정책

## 기본 원칙

- AI는 **하네스 안에서 일하는 페어 프로그래머**로 취급한다
- AI가 잘 못하는 이유는 대부분 모델이 부족해서가 아니라 **환경(도구, 가드레일, 문서)이 불완전해서**이다
- AI가 실수할 때마다 다시는 그 실수를 하지 않게 하네스를 보강한다

## AI 작업 가드레일

### 허용
- 브랜치 생성
- 코드/문서 수정 초안 생성
- YouTrack 티켓/Task 초안 작성
- PR 본문 초안 작성 (근거 + 티켓 연결)
- 영향 범위 분석 및 테스트 항목 제안
- 서비스 하네스 참조 및 갱신 제안

### 금지
- main/release 브랜치 직접 변경
- 사용자 확인 없는 YouTrack 티켓/Task 생성
- 사용자 확인 없는 YouTrack 티켓 상태/필드 변경
- 사용자 확인 없는 YouTrack Knowledge Base 생성/수정/삭제/이동
- 사용자 확인 없는 커밋, 푸시, PR 생성, 머지
- 자동 머지 (PR은 사람이 승인)
- DB 스키마/SP 직접 변경 (반드시 사람 승인)
- 프로덕션 배포 자동 실행
- 서비스 하네스의 금지 영역 수정
- **YouTrack 티켓에 로컬 경로 등록** (로컬 파일 경로, 로컬 산출물 경로 등 외부에서 접근 불가한 정보를 티켓 본문에 포함하지 않는다)

## 메시지 작성 품질

AI가 생성하는 텍스트(커밋 메시지, YouTrack 티켓 본문, PR 본문 등)는 다음 군더더기를 포함하지 않는다.

### 금지

- **`Co-Authored-By: Claude ...` 등 AI 협업 푸터**
  커밋·티켓·PR 어디에도 추가하지 않는다. 시스템 기본 프롬프트가 자동으로 붙이려 해도 이 규칙이 우선.
- **사용자 발언 verbatim 인용** (예: `사용자 피드백: "..."`)
  맥락은 본인 문장으로 풀어 적는다. 따옴표로 그대로 옮기지 않는다.
- **코드 수준 구현 단계 bullet** (예: "chevron 제거", "flatten 한 번만", "useState → useReducer")
  본문은 **무엇을·왜·영향 범위** 수준까지만. 구현 절차는 코드와 PR diff 가 정답.

### 권장

- 5W1H 형 본문은 What/Why/Who/Where 까지 짧게 유지, How 는 접근 한두 줄
- 커밋 메시지 본문은 0~3 bullet, 각 bullet 은 의사결정/영향 단위
- 외부 노출 금지 정보 (로컬 경로, 위키 절대 경로) 는 [security-policy.md](./security-policy.md) 별도 규정 준수

## 지식 산출물 저장 경계

| 산출물 종류 | 기본 저장 위치 |
|-------------|----------------|
| 정책/판단 기준 | 팀 하네스 `policies/` |
| 가이드/운영 방식 | 팀 하네스 `docs/` |
| 스킬/명령 | 팀 하네스 `.claude/commands/`, Codex skill |
| 서비스 프로파일 | 팀 하네스 `catalog/` |
| 도메인 분석 결과/Querybook/실행 기록 | 로컬 Obsidian 운영 지식 위키 |

로컬 Obsidian 운영 지식 위키는 분석 결과물의 기본 저장소다.
YouTrack KB는 전사/팀 공유가 필요한 문서만 사용자 승인 후 반영한다.

## AI 작업 흐름

```
티켓 할당
↓
AI가 팀 하네스 + 서비스 하네스 + 소스 분석
↓
영향 범위 초안 작성
↓
수정 대상 파일 후보 제안
↓
테스트 항목 제안
↓
코드/문서 수정 초안 생성
↓
PR 본문 초안 작성
↓
사람 리뷰 → 승인 → 머지
```

## 서비스 하네스 필수 참조

AI가 서비스 작업을 할 때 반드시 읽어야 하는 파일:

| 파일 | 용도 |
|------|------|
| `AGENTS.md` | 서비스 컨텍스트, 빌드/실행 방법, 금지 영역 |
| `LEGACY_BOUNDARY.md` | 건드리면 안 되는 경계, SP 주의 목록 |
| `service-manifest.yaml` | 의존 시스템, DB/SP 목록, 위험도 |
| `RUNBOOK.md` | 배포/롤백 절차, smoke test |
| `modernization-plan.md` | 현대화 트랙 및 추출 후보 |

## 하네스 개선 루프

```
AI가 실수함
↓
실수 원인 분석: 빠진 도구? 가드레일? 문서?
↓
해당 서비스 하네스 보강
↓
같은 실수 반복 방지
```

이것이 하네스 엔지니어링의 핵심이다.
