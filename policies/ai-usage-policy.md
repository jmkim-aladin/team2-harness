# AI 사용 정책

## 기본 원칙

- AI는 **하네스 안에서 일하는 페어 프로그래머**로 취급한다
- AI가 잘 못하는 이유는 대부분 모델이 부족해서가 아니라 **환경(도구, 가드레일, 문서)이 불완전해서**이다
- AI가 실수할 때마다 다시는 그 실수를 하지 않게 하네스를 보강한다

## AI 작업 가드레일

### 허용
- 브랜치 생성
- 코드/문서 수정 초안 생성
- PR 본문 초안 작성 (근거 + 티켓 연결)
- 영향 범위 분석 및 테스트 항목 제안
- 서비스 하네스 참조 및 갱신 제안

### 금지
- main/release 브랜치 직접 변경
- 자동 머지 (PR은 사람이 승인)
- DB 스키마/SP 직접 변경 (반드시 사람 승인)
- 프로덕션 배포 자동 실행
- 서비스 하네스의 금지 영역 수정

## AI 작업 흐름

```
티켓 할당
↓
AI가 팀 하네스 + 서비스 하네스 + 소스 분석
↓
영향 범위 초안 작성
↓
수정 대상 파일 후보 제안
↓
테스트 항목 제안
↓
코드/문서 수정 초안 생성
↓
PR 본문 초안 작성
↓
사람 리뷰 → 승인 → 머지
```

## 서비스 하네스 필수 참조

AI가 서비스 작업을 할 때 반드시 읽어야 하는 파일:

| 파일 | 용도 |
|------|------|
| `AGENTS.md` | 서비스 컨텍스트, 빌드/실행 방법, 금지 영역 |
| `LEGACY_BOUNDARY.md` | 건드리면 안 되는 경계, SP 주의 목록 |
| `service-manifest.yaml` | 의존 시스템, DB/SP 목록, 위험도 |
| `RUNBOOK.md` | 배포/롤백 절차, smoke test |
| `modernization-plan.md` | 현대화 트랙 및 추출 후보 |

## 하네스 개선 루프

```
AI가 실수함
↓
실수 원인 분석: 빠진 도구? 가드레일? 문서?
↓
해당 서비스 하네스 보강
↓
같은 실수 반복 방지
```

이것이 하네스 엔지니어링의 핵심이다.
