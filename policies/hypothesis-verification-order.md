# 가설 검증 순서 정책

티켓·작업·리뷰에서 의문을 만나면 다음 순서로 검증한다. 사람 컨펌을 1차 수단으로 쓰지 않는다.

## 순서

1. **코드 레벨** — 콜그래프(graphify 등)·grep·심볼 인덱스로 진입점, 호출 SP, 테이블, 외부 의존성을 식별. 동적 SQL은 `unresolved.jsonl` 또는 추가 grep으로 보강.
2. **데이터 레벨** — dev/staging DB 읽기 쿼리(SELECT/EXPLAIN/스키마 메타)로 스키마, 행 카운트, 대표 패턴, 룰 존재 여부 확인. 읽기 쿼리는 사전 동의 ([local-credentials-policy.md](./local-credentials-policy.md) §"dev/staging DB 읽기 쿼리: 사전 동의").
3. **사람 컨펌** — 1·2단계로 풀리지 않는 잔여 미확정만 보고자/오너/연관팀에 질의. 묻는 항목과 그 이유를 묶어 한 번에 전달한다.

이 순서를 건너뛰지 않는다.

## 이유

- 본인 작업 비용을 줄이려고 타인 시간을 쓰는 패턴은 팀 전체 throughput을 깎는다.
- 코드/데이터에서 즉시 답이 나오는 질문을 사람에게 던지면 동일 정보를 두 번 만든다 (요청자가 답하려고 다시 코드/데이터를 본다).
- 사람 컨펌은 **비기록 정보** (의도·예외·과거 경위·외부 조정 결과)를 잡는 용도. 기록 가능한 사실 확인용 아님.

## 위반 시 신호

- 첫 응답이 "보고자에게 어디서 차단했는지 확인 요청"인데, 코드/데이터를 안 본 상태.
- 위키 노트 `## 미확정 질문`에 컨펌 항목만 있고 `## 검증 SQL`/`## 진입점 후보`가 비어있음.
- 사람에게 "X 테이블 이름이 뭐죠?" 류 질문 — 메타 쿼리 한 번으로 풀림.

## 사람 컨펌이 적절한 항목

- 정책·우선순위·범위 결정 (작업 vs 보류, 기능 OFF/ON 여부)
- 과거 동일 사례에서 어떤 결정을 했는지 (코드/티켓에 흔적 없을 때)
- 외부 팀/벤더 합의 사항
- 운영 데이터 정의(이 row가 진짜 차단 대상인가? 등) 중 코드 외부 비즈니스 컨텍스트
- 일정·인력 배정

## 적용 스킬

`work-prep`, `ticket`, `code-review`, `data-request`, `weekly-planned`, `weekly-report`, `service-activity`, `capacity-plan`, `harness-optimize` — 미확정 사실을 다루는 모든 ad 스킬.

각 스킬은 본 정책을 인용하며, 자체 흐름에서 1·2단계를 명시적으로 수행한 뒤 잔여 항목만 컨펌으로 보낸다.
