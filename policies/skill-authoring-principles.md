# 스킬 작성 원칙

팀 스킬(`.claude/commands/ad/*`, `.codex/skills/*`)을 새로 만들거나 수정할 때 적용하는 체크리스트.
스킬은 "확률적인 시스템에서 예측 가능성을 짜내는 장치"다 — 매번 같은 답이 아니라, 매번 같은 방식으로 일하게 만드는 것이 목적이다.

출처: Matt Pocock "How to write good skills" 발표 (AgentOS 정리 영상, 2026-07) + [mattpocock/skills](https://github.com/mattpocock/skills) 레포(MIT) 패턴. 팀 실정에 맞게 번안.

## 대전제 — Codex 패리티

모든 팀 스킬은 Claude Code와 Codex 양쪽에서 동일하게 사용 가능해야 한다 (2026-07-16 팀 확정).

- 절차 SoT는 `.claude/commands/ad/{name}.md` 하나. `.codex/skills/ad-{name}/SKILL.md`는 SoT를 런타임에 읽는 **얇은 alias**로 유지 — 내용 복제 금지
- 스킬 신설·개명·삭제 시 Codex alias를 동반 생성/갱신/삭제한다
- Claude 전용 frontmatter(`disable-model-invocation` 등)는 Codex alias가 파싱하지 않으므로 무해해야 하며, Codex 쪽 동작(사용자 `$ad-*` 호출)을 바꾸지 않는다
- 커버리지 검증: `/ad:harness-optimize 스킬` 실행 시 alias 전수 대조

## 체크리스트 4단계

### 1. 트리거 — 누가 부르는가부터 정한다

내용보다 호출 방식을 먼저 결정한다.

| 방식 | 팀 하네스 구현 | 비용 |
|------|---------------|------|
| 사용자 호출 | `/ad:*` 슬래시 명령 | 인지 부하 — 언제 뭘 부를지 사람이 기억 |
| 모델 호출 | CLAUDE.md Skill routing 규칙 | 컨텍스트 부하 — 설명이 매 요청 상주 + 호출 누락 예측불가 |

- 어느 쪽이 정답이 아니라 **부하를 누가 짊어질지의 트레이드오프**. 스킬 생성 시 선택 근거를 남긴다
- 모델 호출로 만들면 "제때 호출되는가"까지 검증 대상이 된다. 확신 없으면 사용자 호출이 기본값
- 사용자 호출 전용으로 확정한 스킬은 frontmatter **`disable-model-invocation: true`** 명시 — 설명이 모델 컨텍스트에서 완전히 빠져 컨텍스트 부하 0. 게시·배포·동기화 등 사이드이펙트 스킬에 특히 적합
- 모델 호출 스킬의 description은 **트리거 문구 목록이 본체**: 유도 단어를 앞세우고, 트리거 브랜치당 한 문구만 (동의어 나열 = 중복), 본문에 이미 있는 정체성 설명은 넣지 않는다

### 2. 구조 — 절차와 참고 자료를 분리한다

- 스킬 본문 = **절차**(에이전트가 밟을 순서) + **참고 자료**(절차를 돕는 정보) 두 종류뿐
- 본문은 최대한 작게. 작을수록 고치기 쉽고, 감사하기 쉽고, 토큰이 준다
- 절차의 각 단계는 **점검 가능한 완료 기준**으로 끝낸다 — "변경 목록 산출"이 아니라 "수정 대상 전부 확인" 식으로, 끝났는지 아닌지 에이전트가 스스로 판별 가능해야. 모호한 완료 기준은 조기 완료(대충 마무리)를 부른다
- 분리 기준: **매번 쓰는 자료는 본문에, 특정 경로에서만 쓰는 자료는 별도 파일로** 빼고 본문엔 "X 필요하면 Y 파일 참조" 한 줄만 남긴다 (예: 템플릿 → `templates/`)

### 3. 유도 — 설명을 늘리지 말고 강한 단어를 심는다

- 에이전트가 지시를 지나치는 건 문장이 부족해서가 아니라 단어가 약해서다. 긴 설명 대신 **모델이 이미 학습한 업계 용어** 하나를 심고 스킬 전체에서 반복한다 (예: "층별로 나눠 만들지 마라" 문단 대신 "vertical slice" 한 단어)
- 검증법: 에이전트 reasoning에 그 단어가 다시 나타나면 작동하는 것. 안 나타나면 더 강한 단어로 교체
- 특정 단계를 항상 대충 넘어가면(예: 질문 단계 건너뛰고 바로 계획 작성) **스킬을 단계별로 쪼개 다음 단계 목표를 숨긴다**. 목표가 보이면 서두르고, 가리면 파고든다
- 분리는 부하를 지불하는 결정: 독립 트리거 단어가 필요하거나 다른 스킬이 불러 써야 할 때만 모델 호출 스킬로 분리(상주 description 비용 발생), 위처럼 단계 건너뛰기가 문제일 때만 순서 분리
- 사용자 인터뷰 구간에는 질문 규율을 심는다: **한 번에 하나씩, 추천안과 함께, 사실은 조회로 해소하고 결정만 질문** (ad:ticket·ad:work-prep 적용 사례)

### 4. 가지치기 — 지워도 같으면 지운다

거대한 스킬은 병이 아니라 증상. 다음 셋을 정기적으로 제거한다:

- **중복** — 같은 내용의 정답 위치는 한 곳뿐
- **퇴적물** — 여러 사람이 추가만 하고 못 지워서 쌓인 죽은 층
- **무동작 문장** — 있어 보이지만 에이전트 행동을 안 바꾸는 문장. **삭제 테스트**로 찾는다: 문단을 지우고 돌려서 결과가 같으면 그 문단은 처음부터 없었던 것 (모델이 원래 그렇게 훈련된 행동을 재지시하는 문장이 대표적)

가지치기는 `ad:harness-optimize` 실행 시 함께 점검한다.

## 참고 패턴 — 2층 구조

스킬이 많아지면 두 층으로 나눈다:

- **위층**: 한 줄짜리 사용자 호출 진입점 (컨텍스트 부하 0)
- **아래층**: 재사용되는 규율 스킬 (모델 호출)
- 규칙: 위층은 아래층만 부를 수 있고, 위층끼리는 부르지 않는다

새 개념 발명보다 고전 용어(TDD, 도메인 모델링, 리팩토링 카탈로그 등) 재사용이 우선 — 모델이 이미 깊게 아는 단어일수록 유도력이 세다.

## 참고 패턴 — 용어집

도메인 용어가 흔들리면 스킬의 유도력도 흔들린다. 용어집 항목 형식:

- 정의 1~2문장 + 다른 용어와의 관계
- **금지 동의어** 목록 — 같은 개념의 표기 분산을 차단 (예: "이슈 트래커"로 통일, "백로그 매니저" 금지)
- **해소된 중의성 기록** — "X가 두 뜻으로 쓰여 혼동됐고 이렇게 정리했다"를 남겨 재발 방지

팀 적용 위치: vault `wiki/glossary/` (형식만 이 패턴을 따른다). 스킬이 쓰는 강한 용어가 팀 고유어라면 용어집에 정의가 있어야 유도가 작동한다.

## 참고 패턴 — 리뷰 2축 분리

검증형 스킬은 서로 다른 실패 축을 병합하지 않는다. 예: 코드 리뷰의 기준 축(컨벤션 준수)과 스펙 축(요구 충족)은 한쪽 통과가 다른 쪽 실패를 가릴 수 있으므로 따로 판정·보고한다 (ad:code-review 적용 사례).
