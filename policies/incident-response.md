# 장애 대응 프로세스

## 장애 등급

| 등급 | 기준 | 대응 시간 |
|------|------|-----------|
| P1 (Critical) | 서비스 전면 장애, 데이터 유실 위험 | 즉시 |
| P2 (Major) | 핵심 기능 장애, 일부 사용자 영향 | 1시간 이내 |
| P3 (Minor) | 비핵심 기능 장애, 우회 가능 | 업무 시간 내 |

## 대응 흐름

```
장애 인지
↓
영향 범위 파악 (서비스 하네스의 의존 시스템 확인)
↓
즉시 대응 (롤백 / 핫픽스 / 트래픽 전환)
↓
YouTrack 장애 티켓 생성
  - 재현 조건
  - 영향 범위 (사용자 수, 데이터 영향)
  - 원인 분석
  - 조치 내역
↓
근본 원인 분석 (5 Whys)
↓
재발 방지책 → 서비스 하네스 갱신
  - AGENTS.md 주의사항 추가
  - RUNBOOK.md 대응 절차 보강
  - smoke test 항목 추가
```

## 사후 리뷰

- P1/P2 장애는 반드시 사후 리뷰 실시
- 리뷰 결과를 서비스 하네스에 반영
- 같은 유형의 장애가 반복되지 않도록 가드레일 추가

## 장애보고서 작성 기준

장애보고서는 장애 처리 후 사실관계와 재발방지책을 남기는 문서다. 책임 추궁보다 운영 재현성, 원인 재검증, 다음 작업자의 실수 방지가 목적이다.

### 작성 시점

- P1/P2 장애는 복구 당일 초안을 작성하고, 1영업일 이내 리뷰를 완료한다.
- P3라도 운영 DB/SP 직접 변경, 고객/사업부 제보, 반복 가능성이 있는 장애는 보고서를 남긴다.
- YouTrack 장애 티켓 또는 후속 점검 티켓이 있으면 보고서에 이슈 ID와 댓글 시간을 같이 남긴다.

### 필수 항목

| 항목 | 작성 기준 |
|------|-----------|
| 작성자 정보 | 작성자, 부서 |
| 작업정보 | 사고일시, 작업자, 사고내용 |
| 사고원인 | 직접 원인과 근본 원인을 분리해서 작성. DB/SP 변경은 변경 대상과 잘못된 값/조건을 명시 |
| 해결과정 | YouTrack 댓글, 상태 변경, 배포/DB 반영, 복구 확인 시간을 KST 기준 타임라인으로 작성 |
| 직전사고 | 동일/유사 장애 이력. 없으면 `동일 장애 없음` 또는 `확인된 이력 없음` |
| 재발방지 | 즉시 조치, 단기 조치, 장기 검토로 나누고 담당/증적을 남김 |

### 작성 원칙

- 날짜와 시간은 `YYYY-MM-DD HH:mm KST` 형식으로 적는다.
- 사용자가 체감한 현상과 시스템 원인을 구분한다.
- 추정은 `추정`으로 표시하고, YouTrack 댓글/로그/쿼리로 확인된 내용만 단정한다.
- DB/SP 변경이 원인이면 별도 승인 필요 항목으로 표시하고, 변경 전후 코드 조각 또는 핵심 조건을 기록한다.
- 복구자는 원작업자와 다를 수 있으므로 `원작업`, `복구 작업`을 구분한다.
- 재발방지는 “주의”로 끝내지 말고 체크리스트, 검증 쿼리, smoke test, KB/하네스 갱신 중 하나 이상의 실행 가능한 조치로 쓴다.

### 월별 운영 DB/SP 작업 체크리스트

멤버십 월말처리처럼 월별 상수와 쿠폰/이벤트 ID를 SP에 직접 반영하는 작업은 아래 항목을 완료 후 종료한다.

- 요청 티켓의 대상월(`YYYYMM`), 쿠폰 nType, event id를 작업 메모에 표로 정리한다.
- SP의 월별 분기 조건에서 대상월이 중복되거나 누락되지 않았는지 확인한다.
- 변경 후 `@IntMonth` 대상월로 실행되는 분기가 기대 nType을 반환하는지 검증한다.
- PC/모바일 등 요청 채널별 smoke test 또는 담당자 테스트 요청 증적을 남긴다.
- 처리 KB(예: REF-A-221)에 체크리스트가 없으면 YouTrack KB 갱신 요청 또는 하네스 보강 항목으로 남긴다.
