# 사업부 커뮤니케이션 정책

> 적용 범위: 사업부, 기획, 운영, CS 등 비개발 이해관계자에게 전달하는 티켓 댓글, 위키 요약, 선택지, 검토 의견 초안
> 이 정책은 특정 서비스나 만권당에 국한하지 않는 개발 2팀 공통 커뮤니케이션 기준이다.

## 원칙

- 사업부가 이해하는 업무 용어를 먼저 사용한다.
- 내부 구현명, 테이블명, 클래스명, API 경로는 사업부용 문장의 주어로 쓰지 않는다.
- 내부 구현명은 필요한 경우에만 괄호나 `내부 근거` 섹션에 둔다.
- 선택지는 2-4개로 정리하고, 각 선택지마다 효과, 리스크, 결정 필요 항목을 같이 적는다.
- "기술적으로 가능"과 "정책/운영 결정이 필요"를 분리해서 쓴다.
- 확정되지 않은 개발 난이도나 일정은 단정하지 않고, 확인이 필요한 전제와 함께 표현한다.
- 로컬 파일 경로, DB 원문, 계정/토큰/개인정보, 내부 로그 전문은 사업부용 댓글에 쓰지 않는다.
- YouTrack 댓글, KB 변경, 상태 변경은 사용자 승인 후에만 수행한다. 로컬 위키의 초안 작성은 승인 없이 가능하다.

## 내부명 번역 기준

사업부용 초안에는 내부명을 아래처럼 먼저 풀어쓴다. 표의 항목은 대표 예시이며, 다른 서비스에서도 같은 방식으로 업무 용어를 먼저 정한다. 내부명은 개발 근거가 필요할 때만 뒤에 둔다.

| 내부명 | 사업부용 표현 |
|---|---|
| `MaxProduct` 같은 상품/권한 내부 모델명 | 이용권 상품 관리 구조, 상품 관리 구조 |
| `AutoPayment` 같은 결제 내부 모델명 | 기존 정기결제 구조, 자동결제 처리 구조 |
| `MaxPass` 같은 서비스별 권한명 | 이용 권한, 이용권 활성화 |
| table, SP, repository, controller | 저장 구조, 처리 로직, 연동 지점 |
| batch, job, worker | 자동 처리 작업 |

예:
- 나쁨: "`MaxProduct`를 타야 정량제를 처리할 수 있습니다."
- 좋음: "첫 달 이용권도 기존 이용권 상품 관리 구조에서 관리해야 이후 정량제와 후속 처리를 같은 기준으로 추적할 수 있습니다."

## 사업부용 코멘트 구조

사업부에 공유할 초안은 아래 순서를 기본으로 한다.

1. 요약: 현재 판단을 2-3문장으로 설명한다.
2. 선택지: A/B/C처럼 비교 가능한 안을 적는다.
3. 추천안: 개발팀 관점의 권장안을 명시한다.
4. 확정 필요 항목: 사업부가 결정해야 하는 정책, 노출, 운영 기준을 분리한다.
5. 공수/일정 영향: 확정 전제와 함께 대략적인 영향만 적는다.
6. 다음 단계: 승인 후 개발팀이 할 일을 적는다.

## 작성 예시

```text
검토 결과, 이번 건은 "첫 달 990원 이용권을 별도로 판매할 수 있느냐"뿐 아니라
"첫 달 이용권이 활성화된 뒤 다음 달부터 기존 정기결제로 자연스럽게 이어지느냐"가 핵심입니다.

선택지는 두 가지입니다.

A. 첫 달 990원 이용권만 먼저 적용
- 장점: 초기 범위가 작습니다.
- 리스크: 다음 달 자동결제 전환, 정량 사용량 추적, 해지/재구매 기준을 별도로 다시 정해야 합니다.

B. 첫 달 이용권과 다음 달 정기결제 전환까지 같이 설계
- 장점: 첫 달 이후 후속 처리와 정량제 관리 기준을 한 번에 맞출 수 있습니다.
- 리스크: 초기 개발 범위가 커지고, 상품/결제/이용권 활성화 기준 확정이 필요합니다.

개발팀 추천은 B입니다. 다만 사업부에서 우선 출시 범위를 줄여야 한다면,
A로 시작하되 다음 달 전환과 정량 추적 기준을 별도 후속 과제로 분리하는 전제가 필요합니다.
```

## 내부 기록 분리

사업부용 초안과 별도로, 로컬 위키나 개발 메모에는 다음 정보를 `내부 근거`로 분리한다.

- 코드 경로, 클래스명, API, SP, 테이블명
- DB 조회 결과 요약
- 기존 구조와 충돌하는 구현 포인트
- 추가 확인이 필요한 담당 시스템 또는 배포 순서
