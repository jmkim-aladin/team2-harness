# Research: Pitfalls

## Critical Pitfalls

| Pitfall | Warning Sign | Prevention |
|---------|--------------|------------|
| Treating repo DTSX as active truth | Excel repo and DTSX candidates do not map 1:1, especially row 15 | Confirm SQL Agent `msdb` enabled job/step/package before implementation |
| Hidden DB side effects | DTSX contains `TRUNCATE`, `DELETE`, `UPDATE`, staging table prep | Classify every DB operation and wrap side effects in adapter/tasklet |
| Memory-heavy rewrite | Full feed code builds a full list before writing | Use DB-side set operations, paging/range reads, streaming writers |
| File contract drift | Generated bytes differ in encoding, delimiter, newline, sort, null handling | Golden-file tests and byte-level comparison |
| Premature delivery modernization | Migration changes FTP/WWW/SMB contract at the same time as DB cutover | Preserve v1 delivery via bridge; modernize after migration |
| Non-idempotent publish | Failed run leaves partial final file visible | Publish temp artifact, validate, then atomically promote/copy/rename |
| No restart boundary | Rerun repeats destructive SP/table staging unexpectedly | Record snapshot/preparation marker or fail fast unless rerunnable |
| Row 15 over-scope | Naver ranking/sales/product feeds are conflated | Defer row 15 implementation until active job owner confirms scope |

## Phase Guardrails

- Phase 1 must not write code before active evidence is locked.
- Phase 2 must define adapter and manifest boundaries before feed-specific work.
- Phase 3 must create comparison rules before shadow run.
- Phase 4 must implement one vertical slice first, not all feeds in parallel.
- Phase 6 cutover requires human approval and rollback readiness.
