## 00\. 프로젝트 방향 전환 (사전 공유)

> 참조: [multi-storefront-platform-direction.md](./multi-storefront-platform-direction.md)

### 프로젝트 정의

> **멀티테넌트 커머스 오케스트레이션 플랫폼**
> 커머스 인프라(바자르·뉴빌링·Naru)를 연동하여, 운영자가 고객군·채널·정책을 조합해 독립적인 커머스 경험을 즉시 구성·운영할 수 있는 플랫폼.

**Supply / Demand 관점:**
- **바자르** = 공급 측 오케스트레이션 (상품 적재, 재고, 주문 처리, 물류)
- **스토어프론트** = 수요 측 오케스트레이션 (고객 경험, 정책, 채널, 테넌트 구성)
- 스토어프론트가 **소유하는 것**: 테넌트 구성, 정책 엔진, 오케스트레이션 로직
- 스토어프론트가 **소유하지 않는 것**: 상품 데이터(바자르), 결제(뉴빌링), 인증 인프라(Naru)

### 핵심 결정

| # | 결정 | 선택 |
|---|------|------|
| D1 | 프로젝트 성격 | 멀티테넌트 커머스 오케스트레이션 플랫폼 |
| D2 | 인증 방식 | 테넌트별 선택 (Naru SSO/OIDC / 알라딘 일반 회원 / 둘 다) |
| D3 | FE 구조 | 단일 Next.js + 테넌트 설정 기반 (MVP). 팀 규모 상 Micro-FE는 과잉 |
| D4 | MVP 범위 | **여전히 B2B 신규 제휴사부터 시작**. 아키텍처만 멀티 스토어프론트 수용 |

### 기존 설계와의 관계

| 기존 설계 | 유효 여부 |
|----------|----------|
| Schema per Tenant | 그대로 유효 (테넌트가 B2B든 B2C든 동일 작동) |
| Naru OIDC 연동 설계 | 그대로 유효 (auth_type='NARU'인 테넌트의 인증 경로) |
| 정책 엔진 (tenant_policy) | 그대로 유효 (policy_type 확장으로 B2C 정책도 수용 가능) |
| 바자르/뉴빌링 연동 | 그대로 유효 |

### 스코프 문서에서 바뀌는 지점

이 문서의 MVP In/제외/추후 판단은 **"B2B 신규 제휴사"를 기준으로 그대로 유효**합니다. 다만 아키텍처 관점에서 "테넌트"를 B2B 제휴사에 국한하지 않고 설계한다는 전제가 추가됩니다.

- `store_type` 필드를 테넌트에 추가 (MVP는 'B2B'만 사용, 나중에 'B2C', 'CATEGORY' 등 확장)
- `auth_type` 필드를 테넌트에 추가 (MVP는 'NARU'만 사용, Phase 2에 'ALADIN' 추가)

## 01-bis. MVP 유저 여정 — 어떤 흐름을 만들 것인가

> 제어 범위(02절)에 들어가기 전에, MVP에서 만들 **전체 유저 여정**을 공유합니다. 각 단계가 어떤 도메인에 매핑되는지 보면, "이 기능이 왜 MVP In인가"가 자연스럽게 이해됩니다.

### 여정 1: 운영자 (내부 운영팀) — "신규 제휴사 몰 생성"

```
Admin 로그인 → 테넌트 생성 → 정책 설정 → 몰 활성화 → 모니터링 → 정산 확인
     │              │              │            │            │           │
  [인증]        [테넌트 관리]   [정책 엔진]    [테넌트]     [주문]     [정산]
```

1. Admin 콘솔 로그인 (Naru 내부 계정)
2. 테넌트(몰) 생성: 기본 정보 입력, store_type 선택, auth_type 설정
3. 정책 설정: 가격 오버레이, 결제수단 제한, 분야/몰타입 제한, 수량 제한
4. 몰 활성화 → 실구매자 접근 가능
5. 주문 현황 모니터링
6. 월 정산 확인 + 리포트 다운로드

### 여정 2: 실구매자 (B2B 제휴사 직원) — "상품 구매"

```
SSO 인증 → 랜딩 → 검색/전시 → 상품 상세(전용가) → 장바구니 → 주문 → 결제 → 배송 추적
    │         │        │              │                │        │       │         │
 [인증]    [전시]  [검색/AI팀]     [상품/가격]        [주문]   [결제]  [결제]   [주문·배송]
```

1. Naru SSO/OIDC 인증 → 전용몰 진입
2. 랜딩 페이지 (테넌트별 설정)
3. 상품 검색 (분야/몰타입 제한 적용) ← **AI팀 검색엔진 의존**
4. 상품 상세 (제휴사 전용 판매가 = 가격 오버레이)
5. 장바구니 → 주문 (수량 제한 적용)
6. 결제 (허용된 결제수단만 노출)
7. 배송 (알라딘 동일 정책)
8. (필요 시) 부분 취소

### 여정 3: 제휴사 관리자 — "내 몰 운영" (MVP 범위 TBD)

```
Admin 로그인 → 정책 조회/수정 → 주문 현황 → 정산 리포트
     │              │               │            │
  [Admin 권한]  [정책 엔진]      [주문]       [정산]
```

> **논의 필요**: 제휴사 관리자의 MVP 권한 범위. 운영자가 모든 설정을 하고, 제휴사 관리자는 조회만? 아니면 정책 세부 설정(2-tier 모델)까지?

## 02\. 제어 범위 확정 — 어떻게

> 현행 신규 제휴 요청 체크리스트 항목을 기준으로, 사전 확인 결과(반복 요청 빈도·현재 처리 방식)를 반영하여 MVP 포함 여부를 결정합니다. 이견 없는 항목은 빠르게 넘기고, 논쟁이 있는 항목에 시간을 씁니다.

### 분류 기준

* **설정으로 해결 (MVP In)**: 반복 요청이 많고, 이것 없이는 파일럿 운영이 불가능한 것
* **MVP 제외**: 요청은 있었지만 없어도 패턴화하기 어려운 것
* **추후 검토**: 방향은 맞지만 지금 당장 결정하기 어려운 것. **반드시 트리거 명시** (아래 참조)

### "추후 검토"의 Phase 정의 — 트리거 기반

"추후 검토"는 추상적인 "나중에"가 아니라 **"언제 다시 꺼낼 것인가"**를 명확히 한다. 아래 3개 트리거 중 하나를 반드시 표시:

| 트리거 | 의미 | 예시 |
|--------|------|------|
| **[트리거 A] 스토어프론트 확장 시** | MVP 대상이 확장되는 시점 (대량구매몰, B2C 카테고리몰 등 추가) | 세금계산서, 모바일 지원, B2C 정책 |
| **[트리거 B] 제휴사/고객 요구 시** | 파일럿 운영 중 특정 고객이 실제로 요구할 때 | 마일리지, 적립금, 외부 포인트, 선불 충전 |
| **[트리거 C] 의존성 해소 시** | 외부 시스템(바자르/뉴빌링/AI 검색엔진)의 지원이 확인되면 | 배송 제한(바자르 API), 할인율 상품 단위(성능 검증) |

> **왜 이렇게 나누는가**: "추후 검토"를 한 덩어리로 두면 6개월 후 아무도 다시 안 본다. 트리거가 명시되면 해당 이벤트가 발생할 때 자동으로 다시 검토 테이블에 올라온다. 기술 부채와 비슷한 원리.

### Phase 체계와의 관계

이 문서의 "추후 검토"는 **축 4: 기능 성숙도 Phase**에 해당한다. 혼동을 피하기 위해 다른 축과의 관계 정리:

| 축 | Phase 정의 | 이 문서에서의 역할 |
|----|-----------|-------------------|
| 축 1: 기술 아키텍처 | 모놀리스 → DB 샤딩 → MSA → DB per Tenant | 이 문서 범위 밖 (tenant-model.md 참조) |
| 축 2: 프로젝트 전환 | MVP → 안정화 → 기존 제휴사 이관 | 이 문서 범위 밖 (ceo-review.md 참조) |
| 축 3: 스토어프론트 확장 | B2B → 대량구매/카테고리 → B2C → 몰별 FE 분리 | **트리거 A로 연결** (multi-storefront-direction.md 참조) |
| **축 4: 기능 성숙도** | MVP In → 추후 검토 [A/B/C 트리거] | **이 문서의 주요 축** |

> **검색엔진 의존성 (AI 팀 협업 필요)**
>
> 몰타입/분야/검색 제한의 "정책 설정"은 전용몰이 소유하지만, 실제 검색 결과에 반영하려면 AI 팀 검색엔진과의 연동이 필수입니다.
>
> | 구분 | 전용몰 (우리) | 검색엔진 (AI 팀) |
> | --- | --- | --- |
> | 정책 정의 | 제휴사별 허용 몰타입/분야/카테고리를 정책 테이블에 저장 | - |
> | 검색 시 적용 | 검색 요청 시 테넌트 정책을 파라미터로 전달 | 전달받은 필터 조건을 검색 인덱스에 적용하여 결과 반환 |
> | 상품 노출 | 검색엔진 결과 기반으로 전시 | 테넌트별 필터링된 결과 제공 |
> | 구매 제한 | 주문 시 정책 재검증 (검색 우회 방지) | - |
>
> **회의에서 확인할 것:**
> - AI 팀 검색엔진이 **테넌트별 필터 파라미터** (몰타입, 카테고리, 분야)를 지원하는지 또는 지원 예정인지
> - 검색 API 호출 시 필터 조건 전달 방식 (query param? 헤더? 별도 인덱스?)
> - AI 팀 일정과 B2B 전용몰 MVP 일정의 **타임라인 정합성**
> - 검색엔진 미지원 시 **폴백**: 바자르 상품 DB에서 직접 필터링 (성능 이슈 있을 수 있음)
>
> **리스크**: AI 팀 검색엔진 일정이 지연되면 MVP에서 검색 제한이 불완전해질 수 있음. 폴백 전략(DB 직접 필터링) 준비 필요.

## 03\. 구현 도메인 우선순위 — 무엇부터

> 02절에서 MVP In/제외/추후가 확정되면, **구현 순서**를 정합니다. 외부 의존성과 유저 여정 크리티컬 패스 기준.

### 구현 Phase 제안

```
Phase A — Foundation (외부 의존성 없음, 내부 완결)
┌──────────────────────────────────────────────────────────────┐
│  D1 테넌트 관리  +  D2 인증  +  D3 정책 엔진                   │
│  → 이게 없으면 유저 여정 1단계도 안 돌아감                       │
│  → BE 자체 완결 가능, 외부 연동 없음 (Naru 제외)                │
└──────────────────────────────────────────────────────────────┘
          │
          ▼
Phase B — Core Commerce (외부 연동 시작, AI팀 타임라인 맞춤)
┌──────────────────────────────────────────────────────────────┐
│  D4 상품/카탈로그  +  D5 주문                                  │
│  → 바자르 API 연동 핵심                                        │
│  → 검색 제한은 AI팀 검색엔진 타임라인에 맞춤                     │
│  → 폴백: 바자르 DB 직접 필터링 (성능 이슈 가능)                  │
└──────────────────────────────────────────────────────────────┘
          │
          ▼ (Phase B와 병렬 가능)
Phase C — Money (결제/정산)
┌──────────────────────────────────────────────────────────────┐
│  D6 결제  +  D7 정산                                          │
│  → 뉴빌링 연동                                                │
│  → Phase B의 주문 흐름이 어느 정도 잡힌 후 연동                  │
└──────────────────────────────────────────────────────────────┘
          │
          ▼
Phase D — Operations (운영 레이어)
┌──────────────────────────────────────────────────────────────┐
│  D8 Admin 콘솔                                                │
│  → Phase A~C 위에 얹는 운영 UI                                │
│  → FE 중심 작업 (JEH + 외주 FE)                               │
└──────────────────────────────────────────────────────────────┘
```
