## 0. 프로젝트 정의

### 한 줄 정의

> **멀티테넌트 커머스 오케스트레이션 플랫폼**
> 커머스 인프라(바자르·뉴빌링·Naru)를 연동하여, 운영자가 고객군·채널·정책을 조합해 독립적인 커머스 경험을 즉시 구성·운영할 수 있는 플랫폼.

### Supply / Demand 분리

```

  Supply-side (공급 오케스트레이션)          Demand-side (수요 오케스트레이션)
  ┌──────────────────────────┐          ┌──────────────────────────────┐
  │         바자르            │          │        스토어프론트            │
  │                          │          │                              │
  │  · 상품 적재·카탈로그     │             │  · 테넌트 구성·운영           │
  │  · 재고 관리             │             │  · 고객군 × 채널 × 정책 조합  │
  │  · 주문 처리·물류         │◄────────►   │  · 독립 커머스 경험 생성       │
  │  · 마켓플레이스 연동      │              │  · 인증 라우팅               │
  │                          │          │  · 정책 엔진                 │
  └──────────────────────────┘          └──────────────────────────────┘
             │                                     │
        뉴빌링 ◄───────────────────────────────────┘
        (결제·정산 인프라)
```

**스토어프론트가 소유하는 것**: 테넌트 구성, 정책 엔진, 오케스트레이션 로직
**스토어프론트가 소유하지 않는 것**: 상품 데이터(바자르), 결제 처리(뉴빌링), 인증 인프라(Naru)

### 프로젝트명

| 용도 | 이름 |
|------|------|
| 프로젝트명 (영문) | Storefront |
| 프로젝트명 (한글) | 스토어프론트 |
| service_id | `storefront` |
| 기술 정의 | 멀티테넌트 커머스 오케스트레이션 플랫폼 |

## 1. 방향 전환 요약

```
기존: "B2B 제휴사 전용 커머스 플랫폼"
  ↓
변경: "멀티테넌트 커머스 오케스트레이션 플랫폼"
```

| 항목 | 기존 | 변경 |
|------|------|------|
| 프로젝트 성격 | B2B 전용 | 커머스 오케스트레이션 (B2B + B2C + 카테고리몰 + 대량구매몰 등) |
| 플랫폼 역할 | 단독 커머스 서비스 | 커머스 인프라를 연동하는 수요측 오케스트레이터 |
| 테넌트 정의 | 테넌트 = B2B 제휴사 | 테넌트 = 독립 커머스 경험 단위 (고객군·채널·정책의 조합) |
| 인증 | Naru SSO/OIDC only | 테넌트별 선택 (Naru / 알라딘 일반 / 둘 다) |
| FE | B2B 전용 단일 FE | 단일 FE + 테넌트 설정 기반 화면 분기 |
| 고객 범위 | B2B 제휴사 직원 | B2B 직원 + B2C 일반 고객 + 대량구매 담당자 |

## 2. 왜 바뀌었는가

플랫폼의 핵심 가치가 "B2B 제휴사 운영"이 아니라 **"커머스 인프라를 조합해 독립적인 커머스 경험을 즉시 구성"**하는 것이었다. 이 가치는 B2B에 국한되지 않는다.

나오고 있는 구체적 요구:
- 대량 주문 고객 전용 몰 (기업/공공기관)
- 도서, 음반, 외국도서 등 카테고리별 특화 몰
- 주문/결제 도메인별 프론트 분리
- B2C 고객도 몰 단위로 진입 가능

## 3. 핵심 결정 사항

| # | 결정 | 선택 | 근거 |
|---|------|------|------|
| D1 | 프로젝트 성격 | **멀티테넌트 커머스 오케스트레이션 플랫폼** | 커머스 인프라를 연동해 독립 커머스 경험을 구성·운영 |
| D2 | 인증 방식 | **테넌트별 선택** (auth_type = NARU / ALADIN / BOTH) | B2B는 Naru, B2C는 알라딘 일반, 하이브리드도 가능 |
| D3 | FE 구조 | **단일 Next.js + 설정 기반** | MVP 속도. 팀 규모(FE 1명)에서 Micro-FE는 과잉 |
| D4 | FE 확장 전략 | **MVP: 단일, Phase 2+: 몰별 독립 가능** | BE API 경계를 도메인별로 잡으면 나중에 FE 분리 가능 |
| D5 | MVP 타깃 유연성 | **공통 코어 우선 설계** | MVP가 B2B 제휴사일지 대량구매일지 미확정. 둘 다 수용 가능한 공통 코어 먼저 |
| D6 | Admin 권한 모델 필요성 | **필요 (tenant_admin + RBAC)** | 테넌트별 몰 운영 담당자 지정, 도메인별(정산/정책/주문) 권한 분리 필요 |
| D6-bis | Admin 권한 **위치** | **미결정** (Naru vs 스토어프론트) | 4/15 이후 결정 예정. 아래 4절 참조 |

## 3-bis. MVP 타깃 유연성 (공통 코어 우선)

MVP 타깃이 **B2B 제휴사일 수도, 대량구매일 수도** 있다. 아키텍처는 둘 다 수용해야 한다.

### 두 시나리오 비교

| 시나리오 | 테넌트 | 인증 | 핵심 정책 | 주문 워크플로우 |
|---------|--------|------|----------|---------------|
| **A) B2B 제휴사 MVP** | 제휴사 (멀티캠퍼스 등) | Naru SSO/OIDC | 가격 오버레이, 결제 제한 | 장바구니 → 주문 → 뉴빌링 |
| **B) 대량구매 MVP** | 기업/공공기관 | 알라딘 B2B 계정 / 기업 SSO | 협상가, 대량 할인, 계약 조건 | 견적 요청 → 견적서 → 계약 → PO → 주문 → 세금계산서 |

### 공통 코어 (어느 MVP든 필요)

```
1. 테넌트 모델 (Schema per Tenant + store_type)
2. 인증 추상화 (auth_type별 어댑터)
3. 정책 엔진 (policy_type 확장 가능)
4. 상품 오버레이 (바자르 원본 + 테넌트별 오버레이)
5. 주문/결제 기본 (뉴빌링 연동)
6. Admin 권한 모델 (tenant_admin + RBAC) ← D6
```

### 도메인별 특화 (MVP 타깃 확정 시 구현)

| 도메인 | B2B 제휴사 MVP일 때 | 대량구매 MVP일 때 |
|--------|-------------------|------------------|
| 인증 | Naru OIDC 완성 | 기업 SSO 또는 알라딘 B2B 계정 연동 |
| 정책 | 가격 오버레이 엔진 | 협상가 + 계약 조건 엔진 |
| 주문 | 일반 장바구니/주문 | 견적-계약-PO 워크플로우 |
| 정산 | 월 정산 자동화 | 건별 세금계산서 발행 |

**리스크**: 공통 코어만 먼저 만들고 MVP 타깃을 미뤄두면 Q3 개발 일정에 영향 있을 수 있음. 4/15 회의에서 MVP 타깃을 확정하는 것이 이상적.

## 4. Admin 권한 모델 (Naru 기능 강화)

### 문제

Naru 현재 구조는 **엔드 유저 인증**(파트너 직원이 전용몰 로그인)만 다룬다. **몰 운영자/관리자**는 없다.

```
Naru 현재:
  partner (테넌트)
    └── partner_user (엔드 유저 = 실구매자)
         └── partner_user_mapping (외부 ID 매핑)
```

멀티 스토어프론트에서는 이게 치명적이다:
- B2B 제휴사: 해당 제휴사 운영 담당자가 자기 몰의 정책/정산만 관리
- 대량구매몰: 영업팀이 견적/계약, 정산팀이 세금계산서 (역할 분리)
- 카테고리몰: 내부 MD가 운영, 여러 카테고리몰 담당 가능
- 본부 운영자: 모든 몰 총괄

### 필요한 모델 (위치 미결정)

```
tenant_admin (NEW) ← 몰 관리자/운영자
  ├── user_id (알라딘 계정 or Naru 내부 계정)
  ├── tenant_id (어느 몰의 관리자인가)
  ├── roles [] (OWNER, SETTLEMENT_VIEWER, POLICY_EDITOR, ORDER_MANAGER ...)
  └── scope (특정 도메인 vs 전체)

permission_policy (RBAC 정책)
  - role → permissions 매핑 (resource.action 기반)
  - 예: SETTLEMENT_VIEWER → [settlement.read, report.download]

tenant_admin_session (Admin 콘솔 로그인 세션)
```

### 위치 결정: Naru vs 스토어프론트 (미결정)

**이 모델을 어디에 둘지가 중요한 경계 결정입니다.**

| 기준 | Naru에 두기 | 스토어프론트에 두기 |
|------|-----------|-------------------|
| **일관성** | Naru = 모든 사용자/권한의 단일 소스 (엔드 유저 + 관리자) | 스토어프론트 = 몰 운영의 단일 소스 |
| **기존 자산 재사용** | partner, partner_user 모델과 자연스러운 확장 | 테넌트 정책 엔진과 같은 위치 |
| **Naru 부담** | Naru 역할 확대 (인증 게이트웨이 → 인증 + 인가 관리) | Naru는 엔드 유저 인증만, 역할 명확 |
| **다른 서비스 확장성** | LMS/만권당 등 다른 서비스도 Naru의 Admin 권한 모델 재사용 가능 | 스토어프론트 전용. 다른 서비스는 자체 Admin 필요 |
| **배포 독립성** | Admin 기능 변경 시 Naru 배포 필요 | 스토어프론트 배포만으로 가능 |
| **일반 B2C 서비스 패턴** | - | 일반적 (각 서비스가 자기 Admin 관리) |
| **알라딘 기존 패턴** | 알라딘 내부 계정 체계와 Naru가 연결되면 일관성 있음 | 기존 알라딘 서비스들은 자체 Admin 사용 중일 가능성 |

### 장단점 요약

**Naru에 두는 경우:**
- 장점: "테넌트에 속한 모든 사람(엔드 유저 + 관리자)"의 단일 관리. 향후 여러 서비스가 공유 가능. 권한 정책이 인증과 같은 위치에 있어 일관성.
- 단점: Naru 역할 확대 → 스코프 크리프 위험. Naru가 bottleneck이 될 수 있음. 서비스별 고유 Role이 생길 때마다 Naru 변경 필요.

**스토어프론트에 두는 경우:**
- 장점: 스토어프론트가 자기 몰의 Admin을 독립적으로 관리. Naru는 인증 게이트웨이 역할 유지. 각 서비스(LMS, 만권당 등)도 자기 Admin을 가지게 되면 일관성.
- 단점: 여러 서비스가 생기면 각자 tenant_admin 테이블 중복. 본부 운영자가 여러 서비스를 관리할 때 단일 로그인/권한이 어려움.

### 결정 기준 (4/15 회의에서 논의)

다음 질문에 답하면 위치가 결정됨:

1. **다른 서비스도 같은 Admin 권한 모델을 쓸 것인가?** (LMS, 만권당, 미래 서비스)
   - Yes → Naru에 두는 것이 DRY
   - No → 스토어프론트에 두는 것이 격리 면에서 좋음

2. **본부 운영자가 여러 몰/서비스를 단일 로그인으로 관리할 것인가?**
   - Yes → Naru에 중앙집중
   - No → 각 서비스 분산

3. **Naru가 인증+인가를 모두 담당할 여유가 있는가?**
   - Yes → Naru 확장
   - No → 스토어프론트에서 시작, 나중에 공통화

### 단기 전략 (위치 결정 전)

위치가 결정될 때까지 **인터페이스를 Port로 정의**해두면, 나중에 어느 쪽에서 구현해도 됨:

```kotlin
// core/admin/ (위치 독립적)
interface TenantAdminRepository {
  suspend fun findByUserId(userId: String, tenantId: String): TenantAdmin?
  suspend fun hasPermission(admin: TenantAdmin, resource: String, action: String): Boolean
}

// MVP: 구현체는 위치 결정 후
// - Naru에 둘 때: NaruTenantAdminAdapter
// - 스토어프론트에 둘 때: StoreTenantAdminAdapter
```

이러면 위치 결정을 MVP 실행 시점까지 미룰 수 있음.

### 예상 역할(Role) 예시

| Role | 권한 | 대상 |
|------|------|------|
| `OWNER` | 모든 권한 | 테넌트 최고 관리자 (1명) |
| `POLICY_EDITOR` | 정책 조회/수정 | 운영 담당자 |
| `SETTLEMENT_VIEWER` | 정산 조회/다운로드 | 정산팀 |
| `ORDER_MANAGER` | 주문 조회/관리 | 주문 담당자 |
| `CONTRACT_MANAGER` | 견적/계약 관리 (대량구매몰 전용) | 영업팀 |
| `PLATFORM_ADMIN` | 모든 테넌트 관리 | 본부 운영자 (내부 운영팀) |

### 엔드 유저 인증 vs 관리자 인증

| 구분 | 엔드 유저 인증 | 관리자 인증 (NEW) |
|------|---------------|-------------------|
| 대상 | 실구매자 | 몰 운영 담당자 |
| 인증 방식 | 테넌트별 (Naru OIDC / 알라딘 일반) | Naru 내부 계정 + RBAC |
| 로그인 목적 | 전용몰에서 상품 구매 | Admin 콘솔에서 몰 운영 |
| 기존 Naru OIDC 설계 | 그대로 유효 | 별개 신규 기능 |

### MVP 포함 여부

| 기능 | MVP In? | 비고 |
|------|---------|------|
| `tenant_admin` 테이블 + 기본 CRUD | **MVP In** | 운영자 없이는 몰 운영 불가 |
| 기본 Role (OWNER, POLICY_EDITOR) | **MVP In** | 최소 2-3개 Role로 시작 |
| 세분화된 RBAC (모든 Role) | **Phase 2** | MVP에서는 enum/하드코딩, Phase 2에서 DB RBAC |
| Admin 콘솔 로그인 (Naru 내부 세션) | **MVP In** | 기본 로그인/세션 관리 |
| 감사 로그 (누가 언제 뭘 바꿨나) | **MVP In** | security_audit_event 확장 |
| SSO로 Admin 로그인 | **Phase 2** | MVP는 Naru 자체 계정 |

## 5. 아키텍처 영향

### 바뀌는 것

```
테넌트 모델:
  기존: tenant_registry.type = 'B2B_PARTNER'
  변경: tenant_registry.store_type = 'B2B' | 'B2C' | 'CATEGORY' | 'BULK_ORDER' | ...

인증 설정:
  기존: 모든 테넌트 → Naru
  변경: tenant_registry.auth_type = 'NARU' | 'ALADIN' | 'BOTH'
        + auth_config (OIDC 설정 또는 알라딘 인증 설정)

정책 범위:
  기존: B2B 정책만 (가격 오버레이, 결제 제한, 분야 제한)
  변경: + B2C 정책 (프로모션, 쿠폰, 일반 배송 등) 추후 추가 가능
```

### 바뀌지 않는 것

| 기존 설계 | 유효 여부 | 이유 |
|----------|----------|------|
| Schema per Tenant | **그대로 유효** | 테넌트가 B2B든 B2C든 스키마 격리는 동일하게 작동 |
| Naru OIDC 연동 (D1~D8) | **그대로 유효** | B2B 테넌트의 인증 경로. B2C가 추가되어도 Naru 설계는 변경 없음 |
| 정책 엔진 (tenant_policy) | **그대로 유효** | policy_type을 확장하면 B2C 정책도 수용 |
| 바자르 상품 연동 | **그대로 유효** | 상품 소스는 몰 유형과 무관 |
| 뉴빌링 결제 연동 | **그대로 유효** | 결제 처리는 몰 유형과 무관 |
| JWT 기반 토큰 전달 | **그대로 유효** | JWT claim에 store_type 추가하면 됨 |
| Caffeine 캐시 30분 | **그대로 유효** | |
| JWKS 엔드포인트 | **그대로 유효** | |

### 새로 필요한 것

| 항목 | 설명 | MVP 포함? |
|------|------|----------|
| `store_type` 필드 | tenant_registry에 몰 유형 구분 | MVP In (enum으로 시작) |
| `auth_type` 필드 | tenant_registry에 인증 방식 선택 | MVP In |
| 알라딘 일반 회원 인증 어댑터 | Naru 외에 알라딘 기존 인증 연동 | B2C 몰 포함 시 MVP In |
| FE 테넌트 설정 기반 렌더링 | 테넌트 설정에 따라 레이아웃/컴포넌트 분기 | MVP In (최소한으로) |
| B2C 정책 타입 확장 | 프로모션, 쿠폰 등 | **추후** (MVP는 B2B 정책만) |

## 6. MVP 범위 재정의

**MVP 원칙은 변경 없음**: 신규 제휴사(B2B)부터 시작. 하지만 아키텍처는 멀티 스토어프론트를 수용하도록 설계.

```
MVP (Q3 2026):
  - store_type = 'B2B' (신규 제휴사)
  - auth_type = 'NARU' (SSO/OIDC)
  - 정책: B2B 정책 (가격, 결제, 분야 제한)
  - FE: 단일 Next.js, B2B 전용 화면

Phase 2:
  - store_type += 'CATEGORY' (카테고리 전문몰)
  - auth_type += 'ALADIN' (알라딘 일반 회원)
  - 정책 += B2C 정책 (프로모션, 쿠폰)
  - FE: 테넌트 설정 기반 화면 분기

Phase 3:
  - store_type += 'BULK_ORDER', 'B2C' 등
  - 몰별 FE 독립 배포 (팀 규모에 따라)
  - 도메인별 정책 엔진 확장
```

## 9. 기술 관점에서 좋은 소식

기존에 Schema per Tenant + 정책 엔진 + Hexagonal Architecture로 설계한 것이 이 방향 전환을 자연스럽게 수용합니다. "B2B 전용몰"을 만들면서 실은 "멀티 스토어프론트 플랫폼"의 코어를 이미 설계하고 있었다는 뜻입니다.

변경 비용이 낮은 이유:
- 테넌트 모델: `store_type` enum 추가 (1줄)
- 인증: `auth_type` 필드 + 알라딘 인증 어댑터 (Phase 2)
- 정책: `policy_type` 확장 (기존 구조 그대로)
- FE: 테넌트 설정 기반 분기 (컴포넌트 레벨)

기존 설계를 버리는 것이 아니라 **확장**하는 것입니다.
