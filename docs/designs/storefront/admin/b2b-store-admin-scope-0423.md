## 1. 역할 정의

MVP에서 등장하는 설정·조회 주체는 3명.

| 역할 | 설명 | 접근 채널 |
|---|---|---|
| **플랫폼 운영자 (내부)** | 알라딘 개발2팀/운영팀. 테넌트 생성, 계약 반영, 가격 정책, 포인트 어댑터 설정 등 플랫폼 공통 설정 담당 | 내부 어드민 |
| **제휴몰 관리자 (테넌트 측)** | 제휴사(또는 대량구매 고객사)의 몰 운영 담당자. 자사 회원 정책·주문 조회·기본 정보 열람 | 제휴몰 어드민 |
| **구매자 (임직원/대량구매)** | 도서를 실제로 구매하는 최종 사용자 | 스토어프론트 |

> 구매자 플로우는 DEV2-A-1087 §2에 이미 정의됨. 이 문서는 **플랫폼 운영자**와 **제휴몰 관리자** 두 주체를 다룬다.

### 3.1 개발 리드 협의 결과

|------|---------------|------|
| 베스트셀러·신간 데이터 출처 | **A안 — 알라딘 기존 B2C 데이터 활용 + 테넌트 노출 범위 필터** | SF 자체 집계 X (Phase 2). 집계 기간 설정 기능은 MVP 제외 |
| 주문 상태 수동 변경 | **A안 — MVP 조회만** | 운영자 개입 케이스는 Phase 2 정의 |
| CS 기능 구축 방식 | **MVP에서 내용 조회·대응 제공** (직접 구축 vs 알라딘 채널 활용은 미정) | CS BC MVP 편입 (D-16 결정). 채널 결정은 후속 |

### 3.2 법적 검토 결과

| 항목 | 결정 | 영향 |
|------|------|------|
| SSO 연동 회원 알림 발송 | **배송 시 입력한 배송정보 + 개인정보로 알림톡 발송** | 알림 BC 설계 — SSO 회원도 거래성 알림 수신 가능. 광고성 알림은 별도 동의 체계 (Phase 2) |

> SSO 연동 회원의 경우 이메일·휴대전화 등 개인정보를 SSO 단계에서 수집하지 않음. 따라서 거래성 알림(주문확인·배송)은 **배송지 입력 시 별도 동의받은 연락처로 발송**한다.

### 5.3 어드민 UI 구조 (4/29 회의 결과 반영)

```
내부 어드민 (플랫폼 운영자)
├─ 테넌트 설정
│  ├─ 기본 정보 (이름·로고)         ← 전용 URL은 Phase 2
│  ├─ 상품 노출 범위 (몰타입·화이트리스트)
│  ├─ 가격 정책 (할인율 기준·몰타입별)

├─ 운영
│  ├─ 전시 (베스트셀러·신간 = 알라딘 B2C + 노출 필터)
│  ├─ 공지·안내 콘텐츠 (P1~P2, MVP 후순위)
│  ├─ 주문 조회·상세
│  ├─ 결제 내역 조회
│  ├─ 클레임 현황 조회
│  ├─ CS (문의 조회·대응)
│  └─ 알림 (거래성 알림 — 배송 정보 기반)
├─ 모니터링 (전체 테넌트 주문현황·주문량)
└─ 플랫폼 레벨 설정
   └─ 배송비 기본 정책 (몰타입별)

제휴몰 어드민 (제휴사 관리자, 테넌트 격리)
├─ 회원
│  ├─ 회원 목록
│  └─ 회원별 구매·배송 제한 정책
├─ 모니터링 (자사 주문현황·주문량)
├─ 주문·취소 내역 (자사 범위)
├─ 알림 종류 on/off
└─ 기본 정보 조회 (계약·할인율·결제수단·정책)
```

### 5.4 권한 모델 영향 (변경 없음)

- 플랫폼 운영자 / 제휴몰 관리자 **2-tier 권한 구분 필수** (MVP)
- 제휴몰 관리자는 **자사 데이터만** 접근 (테넌트 격리)
- 구매자 본인 주문·알림은 스토어프론트 권한
