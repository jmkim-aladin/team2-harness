# B2B 스토어프론트 MVP 합의 — 4/22

> **목적**: 첫 라이브에 꼭 필요한 기능을 오늘 같이 확정한다.
> **참석**: 기획자 + 김정민
> **소요**: 30분
> **상태**: 협의 진행 중

---

## 오늘 결정할 것 (딱 3가지)

1. **첫 고객을 누구로 할지** 확정
2. **꼭 필요한 기능 목록** 합의
3. **지금은 안 하지만 설계에서 고려할 것** 합의

---

## 1. 첫 고객

**제휴사 임직원 (도서 구매자)**

예시: 삼성전자 DS·LG생활건강 같은 제휴사의 **임직원**이 몰에 접속해 도서를 직접 구매.

### 성공 정의 (한 문장)

> "임직원이 도서 N권을 주문해서 받고, 필요하면 취소할 수 있다."

### 오늘 빼는 것

| 대상 | 언제 |
|------|------|
| 견적 고객 (공공기관·학교·일반기업) | 내년 1분기 |
| 제휴사 운영팀 (매출·세금 관리) | 내년 이후 |

**왜 임직원부터?** 비회원 인증·조직 인증·매칭·후불 정산이 없어서 가장 빠르게 라이브 가능.

---

## 2. 꼭 필요한 기능 (Must)

임직원이 도서 1권을 주문-받기-취소까지 하려면 반드시 있어야 하는 것.

```
① 로그인 → ② 상품 찾기 → ③ 상품 보기 → ④ 장바구니 →
⑤ 주문 → ⑥ 결제 → ⑦ 배송 확인 → ⑧ 취소(필요 시)
```

| # | 기능 | MVP 범위 (축소) | MVP |
|---|------|---------------|---------------|
| 1 | **로그인·권한** | Naru 로그인만 | O |
| 2 | **테넌트 관리** | 1개 테넌트 기본 구조 | O |
| 3 | **상품 카탈로그** | 공급시스템 연동 + 가격 표시만 | O |
| 4 | **가격** | 정가(판매가) + 테넌트별 고정 할인율 | O |
| 5 | **재고** | 공급시스템 실시간 조회 (캐시 없이) - 바자르 | O |
| 6 | **주문** | 주문서 생성·정책 검증 | O |
| 7 | **결제** | 뉴빌링 단건결제 + 제휴 포인트 단건 = 복합결제 (뉴빌링 제공 수단) - 첫 고객사에 따라 결정 | O |
| 8 | **배송** | 상태 추적 (일단 단일 배송지) -  첫 고객사에 따라 결정 (멀티배송지) | O |

## 3. 있으면 좋은 것 (Should — 대체로 포함)

없으면 사용자가 항의할 수 있는 수준.

| # | 기능 | MVP 범위 (축소) | MVP |
|---|------|---------------|---------------|
| 9 | **장바구니** | 세션 기반 단순 장바구니 - 필수 | O |
| 10 | **취소** | **취소/부분취소만 필수** (반품·교환은 나중에) - 수동대응 - next | 취소/부분취소 |
| 11 | **검색** | 알리스 단순 검색 - 필수 (상품 카탈로그 기반 필터) - 12번 연계 | O |
| 12 | **전시** | 기본 랜딩 + 카테고리 + 큐레이션, 베스트셀러 - 필수 (최소) | O |
| 13 | **알림** | 알라딘 기존 알림 시스템 연동 (주문확인·배송) - 검토 - 법적리스크 여부로 필수 여부 결정 | 법적 리스크 검토 후 필수인경우만 O |

---

## 4. 나중에 할 것 (Could — 축소해서 포함 or 수동 대응)

기능 자체는 있지만 **최소한**으로 시작.

| # | 기능 | MVP는 어떻게? | 나중에 (Phase 2+) | MVP |
|---|------|--------------|---------------------|---------------------|
| 14 | **혜택** | 알라딘 포인트 연동만 (사용-적립 MVP 제외) | 제휴 포인트 원장·쿠폰 | X |
| 15 | **회원·조직** | Naru 로그인 + 조직 유형 필드 예약 | 조직 수기 인증 (무시) | X |
| 16 | **정산** (플랫폼 ↔ 제휴사) | 수동 CSV export - x2 | 자동 집계·상태 머신 | X |
| 17 | **CS·고객 응대** | 이메일·전화로 수동 대응 - next | SF 내장 CS 도구 | X |

## 5. 지금은 안 함 (Won't — 설계 훅만)

**기능은 완전히 빼지 않고**, 스키마 컬럼·인터페이스만 미리 만들어 둔다. 실제 구현은 다음 Phase.

| 대상 | 이유 | MVP |
|------|------|------|
| 견적 몰 (견적 요청·매칭·견적서·후불 정산 등 7개) | 견적 고객은 2번째 타깃 | X |
| 리뷰 | 임직원 채널이라 우선순위 낮음 | X |
| 구매 모니터링 (통계·허용 불가 품목 감지) ***주문현황, 주문량 (슈프림 슈쿠드) admin*** | 제휴사 운영팀용, 지금 타깃 아님 | O |
| 계약 관리 UI | MVP는 하드코딩, 화면은 나중에 | X |
| 제휴사 전용 URL·SSO - 대형 고객사인경우만 고려 | 확장 기능 | X |

## 6. 기획자에게 꼭 확인할 것 (5가지)

### ① 첫 고객 합의

- 임직원 먼저 가는 것 맞나요? -> 이번주 (17주차) 결정 될듯
- 빠진 타깃(견적·운영팀)은 언제 필요한가요? **"6개월/1년/2년 안"** 중 선택.

### ② 구매 품의·승인 워크플로우

- 임직원이 뭔가를 사려면 **상사 승인**이 필요한가요? -> ***회원 별로 구매, 배송 제한??***
- 필요하면 MVP에 Must로 들어가야 함 (설계 맵에서 누락 확인됨).

### ③ 혜택 범위

- 알라딘 포인트 연동만 MVP에 넣어도 되나요? ***아뇨***
- 제휴사 전용 포인트·쿠폰이 **계약 필수**인 곳 있나요? ***있지만 단건***

### ④ 회원·조직 인증

- 임직원은 Naru 로그인만으로 충분한가요?
- 수기 조직 인증이 1차 라이브에 꼭 필요한가요? - 추후

### ⑤ 모니터링

- 제휴사 운영팀이 **구매 통계·허용 불가 품목 감지**를 MVP에 요구하고 있나요? - **구매 정도**
- Phase 2에 밀어도 되나요? 아니

---

## 7. 협의 프레임 (싸우지 않기 위한 방법)

**"이거 빼도 돼요?"** → 저항 유발. 쓰지 말 것.

**"이거 언제까지 필요해요?"** → 시간 차원으로 되묻기.

- **6개월 안** → Must / Should
- **1년 안** → Could
- **2년 안** → Won't (설계 훅만)

현재까지 그린 모든 기능은 **"다 지키되, 언제 할지만 결정"**.
