Hermes + DEV2 하네스 운영 루프 요청 처리, 서비스 지식, 강화되는 에이전트 프로세스
작성 기준 2026-06-12
team2 repo + Obsidian vault

Hermes는 실행을 빠르게 하고, 하네스와 위키가 팀 지식을 고정한다.

전체 구조의 핵심은 저장 경계다. Hermes의 세션, 메모리, 자동 스킬은 개인 런타임 상태이고, 서비스 분석과 요청 처리 결과는 DEV2 운영 위키와 팀 하네스의 정해진 위치로 승격될 때 팀 지식이 된다.

운영 원칙

요청은 YouTrack 5W1H 또는 회의/운영 맥락에서 시작한다.
분석 결과는 vault의 서비스·티켓 구조에 맞춰 draft로 저장한다.
정책, 스킬, 카탈로그는 team2 repo가 source of truth다.
티켓 변경, KB 변경, 커밋, 푸시, PR은 사람 승인 후 실행한다.

3. Hermes의 강화 루프

강점은 “대화 기억” 자체가 아니라, 반복 작업에서 절차를 추출하고 다음 작업의 초기 맥락 품질을 높이는 것이다.

관찰요청, 레포 구조, 서비스 하네스, 기존 vault 노트를 읽는다.
패턴화반복되는 조회 경로, 검증 순서, 위험 신호를 작업 패턴으로 묶는다.
초안화티켓 prep, 분석 노트, PR 본문, 테스트 계획을 빠르게 만든다.
검증테스트, 코드 근거, 사용자 확인, 리뷰로 추론을 걸러낸다.
승격검증된 지식만 vault canonical 또는 하네스 정책으로 옮긴다.
재사용다음 요청에서 더 짧은 맥락 탐색과 더 정확한 체크리스트로 시작한다.

개인 런타임과 팀 지식은 분리한다

Hermes의 sessions, memories, skills는 빠른 실행을 돕는 개인 상태다. 팀 기준으로 남길 내용은 vault frontmatter와 하네스 정책 경계를 통과해야 한다.

4. 저장 경계

같은 분석 결과라도 어디에 저장하느냐에 따라 의미가 달라진다. 이 경계를 지켜야 Hermes 결과물이 팀 운영 지식과 충돌하지 않는다.

개인 런타임

Hermes profile

대화, 임시 기억, 자동 스킬, 작업 캐시.

  • ~/.hermes/profiles/team2
  • 팀 canonical 지식 아님
  • 초안 생성과 반복 실행에 사용
업무 내용

Obsidian vault

도메인 분석, 티켓 산출물, 회의록, 결정 후보.

  • wiki/services/{service}
  • wiki/processes/tickets
  • frontmatter로 관계 연결
일하는 법

team2 repo

정책, 스킬, 카탈로그, 운영 가이드.

  • policies/
  • docs/
  • .codex/skills/
코드 원본

Service repo

구현, 테스트, OpenAPI, 배포와 강결합된 문서.

  • AGENTS.md
  • LEGACY_BOUNDARY.md
  • RUNBOOK.md
공유 원천

YouTrack KB

전사·팀 공통 지식. 변경은 사람 승인 후 REST API로만.

  • 전사 보안/개발 규칙
  • 팀 공통 온보딩
  • 하네스는 링크와 매핑 유지

5. 실제 요청 처리 예시

서비스 분석 요청이 들어왔을 때 어느 지식이 읽히고, 무엇이 어디에 남는지 한 번에 보는 흐름이다.

분석 실행 흐름 Hermes가 빠르게 처리
입력
“max 주문 흐름 분석해줘” 또는 DEV2 티켓 번호.
읽기
team2 정책, max catalog, 서비스 하네스, 관련 vault 노트, 코드 경로.
분석
API, SP, table, UI 흐름의 근거를 source path 중심으로 추적.
초안
영향 범위, 위험, 테스트 후보, 결정 대기 항목을 정리.
지식 반영 흐름 팀 구조에 맞춰 저장
티켓 단위
processes/tickets/in-progress/DEV2-xxxx.md
서비스 분석
services/max/analysis/order-flow-code-review.md
도메인 지식
services/max/domains/order/ 아래 business-rules, traps, relations.
정책 승격
반복 가능한 규칙만 team2 policies/, docs/, skill로 이동.

6. 보안과 승인 게이트

Hermes가 강해질수록 권한을 줄여야 한다. 실행 능력은 sandbox로 제한하고, 팀 상태 변경은 사람 승인으로 잠근다.

항상 잠긴 작업

아래 작업은 Hermes가 초안과 분석까지만 수행하고, 실제 변경은 명시 승인 후 별도 실행한다.

  • YouTrack 티켓/Task 생성, 상태 변경, 필드 변경
  • YouTrack KB 생성, 수정, 삭제, 이동
  • git 커밋, 푸시, 머지, PR 생성/머지
  • DB/SP 변경, 프로덕션 배포
  • 시크릿 조회 결과를 로그, 위키, 티켓에 남기는 행위

권장 Hermes 실행 자세

team2 전용 프로파일, Docker 백엔드, 빈 docker_forward_env, vault read-only mount에서 시작한다. 쓰기가 필요하면 특정 inbox 또는 대상 analysis 경로만 열고, 생성물은 draft로 저장한다.

이 문서는 team2 하네스의 지식 경계와 AI 사용 정책을 시각화한 운영 가이드다. 실제 자동화 도입 시 서비스별 하네스와 보안 정책을 우선 적용한다.
