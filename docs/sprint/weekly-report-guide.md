# 주간업무 보고 가이드

> 출처: [YouTrack KB DEV2-A-692](https://aladincommunication.youtrack.cloud/articles/DEV2-A-692/7KO86rCE7JeF66y0)

## 1. 계획 업무 작성 원칙

기본적으로는 계획된 업무 중 개발자 담당 티켓을 기준으로 작성합니다.

**기록 대상**:

- Type=Feature 또는 Epic
- 개발자 담당 티켓 우선
- 대상 월 스프린트와 일치하는 티켓
  - 예: 2026년 6월 보고서는 `Sprints=2026.06` 또는 `2606-planned`
- 기획자/디자이너 Feature 하위에 개발자 담당 Task가 있으면 부모 Feature를 컨텍스트로 사용하고 개발자 Task만 본문에 표기
- 완료 항목은 대상 월 완료분 유지

**기록 제외**:

- Backlog 상태 티켓
- 대상 월 스프린트가 아닌 티켓
- 기획자/디자이너 담당 Feature 중 개발자 담당 하위 Task가 없는 항목
- 사업부 작성 단발 운영 Task/Bug (통계요청·점검요청·팀장승인·앱푸시 발송 리스트·개별 사용자 문의 등)
- Type=Task/Bug 단발 대응 일반

**예외 포함**:

- 이슈 규모가 크거나 팀 차원의 공유가 필요한 운영성 업무(사업부 작성 티켓이라도 Feature 성격이면 포함 가능)
- 멀티캠퍼스 IF 개발, B2B 플랫폼 구축 등 장기 운영성 Feature

**중복 제거**:

- `DEV2-*` ID 중복은 같은 레벨에서만 제거합니다.
- top-level 항목끼리 같은 ID가 반복되면 하나만 남깁니다.
- 하위 본문 라인끼리 같은 ID가 반복되면 하나만 남깁니다.
- top-level과 하위 본문 라인의 같은 ID 반복은 계층 표현으로 허용합니다. 단일 Task나 하위가 없는 Feature는 제목과 본문 라인에 같은 ID가 반복될 수 있습니다.

## 2. 상태별 구분 기준

### ① 백로그

> 최종 주간업무 보고 후보에서는 Backlog 상태 티켓을 제외합니다. 아래 정의는 과거 양식/조회용 참고입니다.

**정의**

- 현재 진행 중인 스프린트 종료 후 시작 예정인 업무
- 우선순위 변경으로 현재 스프린트에서 제외된 업무

**작성 티켓 기준**

- 티켓 유형: Epic, Feature
- 티켓 상태: Backlog
- 스프린트: 없음

**필수 작성 요소**

- 제목
- 담당자
- 5W1H 포함 티켓 링크

**세부 항목 작성**: 불필요

### ② 미진행 항목

**정의**

- 현재 진행 중인 스프린트에 포함되었으나 아직 시작하지 않은 업무
- 우선순위 조정으로 재정렬된 업무

**작성 기준**

- 티켓 유형: Epic, Feature
- 티켓 상태: Open
- 스프린트: 현재 진행 중인 스프린트
- 보고 대상 월과 스프린트가 일치해야 함 (`YYYY.MM` 또는 `YYMM-planned`)

**필수 작성 요소**

- 제목
- 담당자
- 시작 예상 일자(~)
- 5W1H 포함 티켓 링크

**세부 항목 작성**: 선택

### ③ 진행 중 항목

**정의**

- 현재 스프린트에 포함되어 실제 작업이 진행 중인 업무
- 이전 스프린트에서 이월되어 계속 진행 중인 업무

**작성 기준**

- 티켓 유형: Epic, Feature
- 티켓 상태: In Progress
- 스프린트: 현재 진행 중인 스프린트(이월된 경우 과거 스프린트 이력 포함)
- 보고 대상 월과 스프린트가 일치해야 함 (`YYYY.MM` 또는 `YYMM-planned`)

**필수 작성 요소**

- 제목
- 담당자
- (~)완료 목표 일자
- 5W1H 포함 티켓 링크

**세부 항목 작성**: 필수

### ④ 완료된 항목

**정의**

- 현재 진행 중인 스프린트에 포함되어 완료된 업무
- 이전 스프린트에서 완료되었으나, 지난 보고 이후 완료 처리된 업무

**작성 기준**

- 티켓 유형: Epic, Feature
- 티켓 상태: Fixed (또는 완료 상태)
- 스프린트: 해당 완료 시점의 스프린트 이력 유지
- 보고 대상 월과 스프린트가 일치해야 함 (`YYYY.MM` 또는 `YYMM-planned`)
- 대상 월 완료분은 유지하며 최근 7/14일 범위로 자르지 않음

**필수 작성 요소**

- 제목
- 담당자
- 완료 일자
- 5W1H 포함 티켓 링크

**세부 항목 작성**: 선택

## 3. 세부(Task) 항목 관리 기준

Epic/Feature 하위 Task는 진행 상태를 명확히 구분하기 위해 말머리를 추가합니다.

**말머리 구분**

- (완료)
- (진행 중)
- (예정)

**작성 기준**

- 티켓 유형: Task
- 티켓 상태:
  - (완료) → Fixed
  - (진행 중) → In Progress
  - (예정) → Open
- 스프린트: 현재 진행 중인 스프린트
- 기획자/디자이너 Feature 아래 Task라도 개발자 담당이면 부모 Feature 컨텍스트 아래 표기 가능
- 기획자/디자이너 담당 Task는 주간업무 개발자 보고 본문에서 제외

**라인 표기 규칙**: 4항(항목 표기 규칙) 참조 — 앞부분 간략 설명, 티켓번호 뒤 YouTrack 티켓 제목 원문 그대로.

## 4. 항목 표기 규칙

KB `DEV2-A-696` 원본 패턴이 source of truth. 가이드와 KB 원본이 다르면 KB 원본을 따른다.

### 4.1 제목 라인 (top-level)

```
* **{제목} ({일정정보, 담당자}** DEV2-xxxx **\[원문제목\])**
```

- 불릿: `*` (하이픈 `-` 아님)
- `**` 짝 2개로 분할 — `**...담당자**` ... **`\[원문제목\])**`
- 원문제목의 대괄호는 `\[` `\]` **escape** (KB 마크다운 렌더링 호환)

### 4.2 본문 라인 (sub-task 또는 단일 본문)

```
  : ({상태}) {간략 설명} ({일정정보}, {담당자} DEV2-yyyy [원문제목])
```

- 2-space indent + `:` + space (불릿 아님)
- 본문 라인 대괄호는 **escape 없이** `[원문제목]` 그대로 (YouTrack 자동 링크 동작)
- 앞 간략 설명은 보고서 흐름상 짧게 요약 (Task 본질만)
- 티켓번호(`DEV2-xxxx`) 뒤에는 YouTrack 티켓 제목을 **수정 없이 그대로** 기재 (대괄호 prefix `[서비스][직군]` 포함)

### 4.3 Feature 하위가 없거나 단일 Task인 경우

본문 라인을 **반드시 작성**하며, 본문 라인의 티켓 정보(`DEV2-xxxx [원문제목]`)는 **제목과 동일**하게 반복한다.

**예시 (단일 본문 — 동일 티켓 정보 반복)**:

```
* **만권당 5월 이벤트 (5월의 책) (5/13, 조은흠** DEV2-5259 **\[만권당\]\[개발\] 5월 이벤트 (5월의 책))**
  : (완료) 5월의 책 디자인/개발 일정 (5/13, 조은흠 DEV2-5259 [만권당][개발] 5월 이벤트 (5월의 책))
```

**예시 (다중 sub-task — 각자 자체 티켓 정보)**:

```
* **\[AASM\] IAM Access Key → Node Role + Self-Assume 마이그레이션 (5/15, 김정민** DEV2-6223 **\[AASM\] IAM Access Key → Node Role + Self-Assume 마이그레이션)**
  : (완료) S3 클라이언트 빌더 신설 + 진입점 제한 (5/15, 김정민 DEV2-6225 [AASM] S3 클라이언트 빌더 신설 + 진입점 제한)
  : (완료) S3 연산 모듈 시그니처 일괄 변경 (5/15, 김정민 DEV2-6226 [AASM] S3 연산 모듈 시그니처 일괄 변경)
```

### 4.4 일자정보 포맷 (3항과 동일)

- 진행 중 → `~완료예상일`
- 예정 → `시작예정일~`
- 완료 → `완료일`
- 백로그(보류) → 일자 생략 가능
- 지연 → `~~기존목표~~ ~수정목표`, 제목 뒤 ` - 지연` 표시, 본문에 지연 사유 기재

### 4.5 예정일 선택과 fallback

대상 월 스프린트에서 아직 시작되지 않았거나 종료되지 않은 티켓(`Open`/`Reopened`/`In Progress`)은 각 Feature·Task 코멘트에 기록된 예정일을 일정정보의 기본값으로 사용합니다. 완료 티켓의 완료일은 이 규칙의 대상이 아닙니다.

**코멘트 해석**:

- 코멘트는 자유 형식이므로 제목·본문·목록 위치나 특정 양식에 한정하지 않고 문맥상 업무 일정인지 판별합니다.
- `Open`/`Reopened`는 시작·착수·진행 예정일을, `In Progress`는 완료·종료·마감 목표일을 선택합니다.
- 행사 기간, 배포일, 다음 정기 변경일처럼 티켓 작업의 시작·완료와 무관한 날짜는 제외합니다.
- 취소선 처리된 과거 일정은 무시하고, `updated`(없으면 `created`) 기준 최신 코멘트의 유효한 일정 표현을 우선합니다.
- 범위는 미착수 티켓이면 시작 경계, 진행중 티켓이면 종료 경계를 사용합니다. 상대 날짜는 코멘트 작성 시각의 KST 기준으로 한 날짜가 확정될 때만 변환합니다.
- 추출한 날짜는 `M/D`로 정규화한 뒤 미착수는 `M/D~`, 진행중은 `~M/D`로 표시합니다.

**Feature·Task fallback**:

| Feature 코멘트 명시일 | Task 코멘트 명시일 | Feature 일정정보 | Task 일정정보 |
|---|---|---|---|
| 있음 | 있음 | Feature 자체 날짜 | Task 자체 날짜 |
| 있음 | 없음 | Feature 자체 날짜 | Feature 명시일 상속 |
| 없음 | 하나 이상 있음 | 대상 Task 명시일 중 가장 빠른 날짜 | 날짜가 있는 Task는 자체 날짜 |
| 없음 | 없음 | `<font color="red">미기록</font>` | `<font color="red">미기록</font>` |

- 일부 Task에만 날짜가 있으면 Feature는 기록된 Task 날짜 중 최솟값을 사용하지만, 날짜가 없는 다른 Task는 `<font color="red">미기록</font>`으로 표시합니다.
- Task에서 롤업한 Feature 파생 날짜는 다른 Task로 다시 상속하지 않습니다. Task는 Feature 코멘트의 **명시일**만 상속할 수 있습니다.
- fallback 날짜는 표시 대상 티켓의 상태에 맞춰 `M/D~` 또는 `~M/D`로 렌더링합니다.
- 어느 코멘트에서도 유효한 날짜를 찾지 못하면 임의로 추정하지 않고 일정정보 자리 전체를 붉은색 `<font color="red">미기록</font>`으로 표시합니다.

## 5. 템플릿

섹션 헤더는 H2 + bold (`## **백로그 항목**` 식). KB 원본 패턴 준수.

### 1) 백로그 항목

최종 주간보고 후보에서는 Backlog 상태 티켓을 제외한다. 아래 템플릿은 기존 KB 양식 조회·호환용이다.

```
* **{소제목} ({담당자}** DEV2-xxxx **\[원문제목\])**
  : (예정) {본문} ({담당자} DEV2-xxxx [원문제목])
```

### 2) 계획(미진행) 항목

```
* **{소제목} ({시작예정일}~, {담당자}** DEV2-xxxx **\[원문제목\])**
  : (예정) {본문} ({시작예정일}~, {담당자} DEV2-xxxx [원문제목])
```

유효한 자체 날짜나 fallback 날짜가 없으면 `{시작예정일}~` 대신 `<font color="red">미기록</font>`을 사용한다.

### 3) 진행 중 항목

```
* **{소제목} (~{완료목표일}, {담당자}** DEV2-xxxx **\[원문제목\])**
  : (완료) {본문} ({완료일자}, {담당자} DEV2-yyyy [원문제목])
  : (진행 중) {본문} (~{완료목표일}, {담당자} DEV2-yyyy [원문제목])
  : (예정) {본문} ({시작예정일}~, {담당자} DEV2-yyyy [원문제목])
```

유효한 자체 날짜나 fallback 날짜가 없으면 `~{완료목표일}` 또는 `{시작예정일}~` 대신 `<font color="red">미기록</font>`을 사용한다.

지연:

```
* **{소제목} (~~{기존목표}~~ ~{수정목표}, {담당자}** DEV2-xxxx **\[원문제목\]) - 지연**
  : (진행 중) {본문} (~{수정목표}, {담당자} DEV2-xxxx [원문제목])
  : 지연 사유: {사유}
```

### 4) 완료된 항목

```
* **{소제목} ({완료일자}, {담당자}** DEV2-xxxx **\[원문제목\])**
  : (완료) {본문} ({완료일자}, {담당자} DEV2-yyyy [원문제목])
```

지연 완료:

```
  : (완료) {본문} ({완료일자}, {담당자} DEV2-xxxx [원문제목])
  : 지연 사유: {사유}
```

