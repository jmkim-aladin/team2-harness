# 스프린트 워크플로우 실행 가이드

> YouTrack KB: DEV2-A-1044 | 상위: DEV2-A-1045 (운영 방안)

## 1. 티켓 상태 및 관리 기준

모든 작업 티켓은 아래 4가지 상태를 기준으로 운영하며, **스토리 포인트**와 **시작 일자**를 통해 우선순위를 관리합니다.

| 상태 | 필수 이행 및 관리 규칙 |
|------|----------------------|
| **백로그 (Backlog)** | 이번 달 범위 제외 항목. 요청 접수 시 판단하여 후보군으로 관리. 오픈 전 하위 티켓을 생성하고 **스토리 포인트** 기입 |
| **오픈 (Open)** | 백로그 중 당월 소화 가능한 티켓만 상태 전환. **[필수] 예상 시작 일자** 기입 |
| **진행 (In Progress)** | 작업 착수 상태. 개인별 동시 작업 Task는 **3개 이하**로 유지 |
| **종료 (Done)** | **[필수] 결과물 링크 첨부** (문서: YouTrack KB 링크, 디자인 등 / 소스 수정은 커밋 내역으로 대체) |

### 스토리 포인트 기입 규칙

- **최하위 노드 티켓 (Task 등):** 실제 공수를 산정하여 포인트 기입 (백로그 단계에서 수행)
- **상위 노드 티켓:** 스토리 포인트 **'0'**으로 설정 (데이터 중복 합산 방지)

> 상세 산정 기준: [스토리 포인트 가이드](./story-point-guide.md)

## 2. 단계별 실행 프로세스

### Step 1. 요청 접수 및 공수 산정 (백로그 단계)

1. **판단:** 신규 요청 시 "당월 계획" 포함 여부 1차 판단 (불가 시 백로그 유지)
2. **대상 선정:** 백로그 중 다음 달 진행 후보 혹은 우선순위 확정이 필요한 대상 선정
3. **구체화:** 선정된 대상에 한하여 하위 티켓(Task)을 생성하고 **스토리 포인트 기입**
   - 백로그를 효율적으로 관리하기 위해 우선순위 검토 단계에서 분업화 진행
4. **우선순위 확정:** 산정된 포인트를 바탕으로 최종 진행 대상 확정

### Step 2. 스프린트 계획 및 오픈

1. **상태 전환:** 해당 스프린트에 진행할 티켓을 **[Open]**으로 변경
2. **일정 수립:** 모든 Open 티켓에 **예상 시작 일자**를 입력하여 작업 순서 정립

> 월간 배분 구조: [스프린트 계획 운영 개요](./sprint-planning-overview.md)

### Step 3. 작업 진행 및 유연한 계획 변경 (스프린트 중)

1. **착수:** 작업 시작 시 상태를 **[In Progress]**로 변경
2. **계획 변경 (팀장 사전 협의 필수):** 아래 상황 발생 시 반드시 팀장과 협의 후 조치하며, **결정 사유를 티켓 코멘트로 기록**
   - **우선순위 변경:** 긴급 이슈 대응 등으로 기존 작업의 순서나 일정 조정이 필요한 경우
   - **피처 분리:** 업무 범위가 방대해져 당월 내 완료 가능한 부분만 분리해야 하는 경우
   - **이월:** 우선순위 밀림 혹은 일정 부족으로 미완료 Task를 다음 달로 넘겨야 하는 경우
3. **이관 처리:** 이월 확정 시 **이월 코멘트** 작성 후 **[Backlog]** 상태로 전환
   - 이월 코멘트 필수 요소: 사유, 재편성 여부, 재확인 조건 및 시점

> 상세 절차: [업무 계획 변경 절차](./plan-change-process.md)

### Step 4. 업무 종료 및 자산화

1. **검증:** 작업 내용이 5W1H 및 요구사항에 부합하는지 확인
2. **기록:** 상태를 **[종료]**로 변경하고, 티켓 내에 **최종 작업 결과물 링크**를 반드시 첨부 (소스 수정은 커밋으로 증빙)

> 티켓 작성 규칙: [티켓 작성 및 운영 가이드](./ticket-guide.md)
