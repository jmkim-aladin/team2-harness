# 스토리 포인트 가이드 및 운영 방안

> YouTrack KB: DEV2-A-892 | 상위: DEV2-A-785 (Sprints) | 도입 전략: DEV2-A-990 (개발) · DEV2-A-969 (기획 4월 실험)

## 1. 스토리 포인트(Story Point)의 정의

스토리 포인트는 작업에 소요되는 '시간'이 아니라, **작업의 상대적인 크기(Size)**를 나타냅니다.

- **판단 기준:** 복잡도(Complexity), 불확실성(Uncertainty), 작업량(Effort)
- **핵심 철학:** "이 작업이 며칠 걸리는가?"가 아니라 **"이 작업이 우리 팀에게 얼마나 큰 작업인가?"**를 정의하는 것

### 스토리 포인트를 도입하는 이유

1. **개인차 극복:** 숙련도에 따라 걸리는 시간은 다르지만, 작업 자체의 객관적 복잡도는 동일
2. **팀 속도(Velocity) 파악:** 우리 팀이 한 스프린트에 처리 가능한 평균 포인트를 파악하여 무리한 일정을 방지
3. **인식 동기화:** 산정 과정에서 팀장과 담당자 간의 요구사항 이해 차이를 사전에 발견
4. **현실적 계획:** 리스크와 불확실성을 수치에 포함하여 버퍼 없는 촉박한 계획을 예방

## 2. 산정 요소

산정 시 아래 세 가지 요소 중 하나라도 높다면 포인트를 상향 조정하여 판단합니다.

| 요소 | 설명 |
|------|------|
| **복잡도** | 비즈니스 로직이 얼마나 꼬여있는가? 예외 케이스가 많은가? |
| **불확실성** | 요구사항이 명확한가? 처음 시도해보는 기술이나 정책인가? |
| **작업량** | 단순 반복이라도 영향 범위가 넓거나 물리적인 작업량이 많은가? |

### 서비스 성격에 따른 직군별 역할 차이

| 구분 | 디자이너 역할 | 개발자(FE) 역할 |
|------|-------------|----------------|
| React 웹 서비스 | 피그마(Figma) UI/UX 디자인 | 디자인 가이드 기반 컴포넌트 및 로직 구현 |
| **레거시 서비스** | 디자인 + **퍼블리싱(HTML/CSS)** | 디자인/퍼블리싱 파일 기반 로직 연결 및 운영 |

## 3. 직군별 상세 산정 기준표 (1 ~ 13)

### 1) 개발자

구현 난이도, 데이터 변화, 기존 코드에 미치는 영향(Risk)을 종합 고려합니다.

| 포인트 (사이즈) | 로직 및 기능 구현 (Logic) | 데이터 및 인터페이스 (Data/API) | 영향도 및 리스크 (Risk) |
|----------------|--------------------------|-------------------------------|----------------------|
| **1 (XS)** | 단순 텍스트/컬러 변경, 상수 수정 | 기 구축된 환경변수 수정 | 영향 없음, 즉시 배포 가능 |
| **2 (S)** | 단순 분기 추가, 유사 기능 복사 수준 | API 단순 필드 추가, 단순 조회 | 영향 범위 국소적, 테스트 용이 |
| **3 (M)** | **[표준]** 신규 로직, 표준 CRUD 개발 | 신규 API 생성, 단순 테이블 설계 | 연관 모듈 회귀 테스트 필요 |
| **5 (L)** | 복잡한 예외 처리, 다수 상태 관리 | 복잡한 쿼리, 다수 테이블 조인 | 공통 모듈 수정, 높은 결합도 |
| **8 (XL)** | **[분할 권장]** 대규모 리팩토링 | DB 마이그레이션, 대용량 처리 | 장애 시 서비스 중단 리스크 |
| **13 (XXL)** | **[분할 필수]** 아키텍처 변경 | 인프라 교체, 다수 레거시 연동 | 한 스프린트 내 완료 불가 |

### 2) 기획자

정책의 확정성과 타 부서/기능과의 의존성을 중심으로 산정합니다.

| 포인트 (사이즈) | 기획 및 정책 수립 (Policy) | 비고 |
|----------------|--------------------------|------|
| **1 (XS)** | 문구 수정, 명확한 정책 한 줄 추가 | 즉시 결정 가능 |
| **2 (S)** | 기존 기능의 마이너 업데이트 정책 | 내부 검토 후 확정 |
| **3 (M)** | **[표준]** 신규 기능의 상세 UX 설계 | 타 부서 협의 필요 정책 |
| **5 (L)** | 복합적인 비즈니스 로직 설계 | 타 팀과의 데이터 연동 기획 |
| **8 (XL)** | **[분할 권장]** 서비스 도메인의 핵심 개편 | 법적/보안 리스크 검토 필요 |
| **13 (XXL)** | **[분할 필수]** 신규 서비스(모듈) 전체 기획 | 여러 단계의 기획/배포 필요 |

### 3) 디자이너

디자인 시스템 활용도와 결과물의 시각적 복잡도를 기준으로 산정합니다.

| 포인트 (사이즈) | React 웹 서비스 (Figma Only) | 레거시 서비스 (Figma + 퍼블리싱) |
|----------------|----------------------------|-------------------------------|
| **1 (XS)** | 아이콘 교체, 이미지 수정 | 간단한 HTML 태그 및 CSS 속성 수정 |
| **2 (S)** | 기존 컴포넌트 활용 페이지 구성 | 기존 레이아웃 내 정보 추가 및 퍼블리싱 |
| **3 (M)** | **[표준]** 신규 화면/컴포넌트 디자인 | 새로운 페이지 디자인 및 마크업 작업 |
| **5 (L)** | 복잡한 인터랙션 설계 | 레거시 레이아웃 구조 변경 및 퍼블리싱 |
| **8 (XL)** | **[분할 권장]** 섹션 전체 리뉴얼 | 복잡한 폼/인터랙션 포함 마크업 전체 |
| **13 (XXL)** | **[분할 필수]** 대규모 신규 과제 | 레거시 전체 CSS 테마/구조 개편 |

## 4. 협의 및 운영 규칙

### 추정 주체 및 시점

- **사전 산정:** 담당자는 매달 마지막 주 **티켓 회의 24시간 전까지** 본인 담당 티켓에 '초안 포인트'를 YouTrack 전용 필드에 입력
- **확정 시점:** 별도 추정 회의를 두지 않고, 매달 마지막 주 티켓 회의에서 티켓 생성·검토와 함께 진행 (프로젝트별 담당이 분리되어 있어 팀 전체 추정은 정확도를 오히려 낮춤)

### 등록 규칙

- 모든 티켓은 **5W1H** 와 **OKR 연결(태그·링크)** 규칙 준수
- **3점(M) 이상** 포인트 부여 시, 본문(5W1H)에 **포인트 산정의 근거**를 짧게라도 명시 (예: "API 신규 개발 필요", "공통 모듈 수정")
- **스프린트 확정 시** 각 Task 티켓에 **SP 산정 기록 코멘트** 게시 (Feature는 SP 0이므로 제외, 양식은 아래 §4-x)

### SP 산정 코멘트 양식

스프린트 확정 시점에 각 Task 티켓 코멘트로 게시하여 산정 추적성을 확보한다. Feature는 SP 0이므로 제외한다.

```markdown
## SP 산정 기록 (YYYY.MM 스프린트)

**SP [N] / [팀 SP 가이드 §3-1) 개발자 산정표](https://aladincommunication.youtrack.cloud/articles/DEV2-A-892) 적용**

산정 요소 (복잡도·불확실성·작업량):
- 복잡도: [평가]
- 불확실성: [평가, 참조 KB 직접 링크 가능]
- 작업량: [평가]

근거: [한 줄 근거]
대응 구간: SP [N]([XS|S|M|L|XL|XXL]) — "[산정표 구간 설명 인용]"
부모 Feature: [DEV2-XXXX]
```

작성 원칙
- 직군별 산정표(§3-1 개발자, §3-2 기획자, §3-3 디자이너) 중 본인 직군 표 인용
- "복잡도·불확실성·작업량" 세 축으로 평가하되 **날짜 추정·이해관계자 협의 등의 부수 정보는 제외**
- 정책 KB·결정 사항은 **YouTrack KB 직접 링크**로 표기 (예: `[DEV2-A-1109](https://aladincommunication.youtrack.cloud/articles/DEV2-A-1109)`)
- 근거 한 줄은 산정표 구간 매칭 사유만 명시

### 이견 처리 원칙

| 차이 | 처리 |
|------|------|
| 1단계 이내 (예: 3 vs 5) | 담당자 의견 존중 |
| 2단계 이상 (예: 3 vs 8) | 회의 후 1:1 별도 협의 |

### 13점(XXL) 관리 규칙

- **금지 사항:** 13점으로 산정된 티켓은 원칙적으로 스프린트에 그대로 담을 수 없음
- **분할 의무:** 반드시 8점(XL) 이하의 하위 작업들로 쪼개어 다시 산정

### 블로커 처리

- 진행 중 예상치 못한 기술적 복잡도가 증가하면 **YouTrack 댓글로 팀장과 포인트 재협의 요청**
- 협의 후 결정 사유를 코멘트로 남김

### 운영 원칙

- **포인트 = 작업량:** 숙련자가 10분 만에 끝낼 일이라도 객관적 난이도가 높다면 높은 포인트를 부여
- **회고 반영:** 스프린트 종료 후, 실제 소요 시간과 포인트 간의 괴리가 컸던 작업들을 복기하여 기준표를 지속적으로 업데이트

## 5. 단계별 목표 지표 (Success Metrics)

> 출처: DEV2-A-990 §5. 궁극 목표는 **팀장 개입 없이 개발팀이 자율 산정·Velocity 예측하는 구조 확립**.

| Phase | 기간 | 운영 방식 | 핵심 지표 |
|-------|------|----------|----------|
| **Phase 1 — 기준 정착** | 1~2개월 | 담당자 사전 산정 → 티켓 회의 검토 → 확정 | 포인트 부여율 100% / 3점 이상 산정 근거 기록률 100% |
| **Phase 2 — 자율화 전환** | 3~4개월 | 담당자 추정 → 회의 공유 (이견 없으면 자동 확정) | 팀장 재협의 비율 30% 이하 / 전월 대비 Velocity 편차 ±20% 이내 |
| **Phase 3 — 안정화** | 5개월~ | 담당자 추정 → 자율 확정 (이상치 발생 시만 팀장 개입) | 팀장 개입 월 1회 이하 / OKR 직결 티켓 포인트 완료율 90% 이상 |

## 6. 4월 특별 가이드 (실행 데이터 정확도 향상)

> 출처: DEV2-A-990 §3. 4월 Velocity 데이터의 정확도를 높이기 위한 한시 가이드.

- 4월 스프린트에 포함되는 모든 개발 티켓은 가급적 **5점(M) 이하 단위로 쪼개어** YouTrack에 등록 (적극 권장)
- 8점(XL) 이상 티켓은 분할 권장, 13점(XXL)은 분할 필수 (§4 13점 규칙)

## 7. 기획 SP 도입 실험 (4월 한시)

> 출처: DEV2-A-969. 1인 기획자의 자기 객관화 도구로서 SP의 적절성 검증.

- 기획 티켓에 §3-2) 기준으로 포인트 부여
- 스프린트 종료 후 **실제 소요 시간(Hour)** 과 **업무 성격(Task Type)** 기록
- 회고 시 다음을 분석:
  - 산정 포인트 vs 실제 시간 상관도
  - 기존 기준(정책 확정성·의존성) 외 추가 보정 기준의 필요성 (소요 시간·업무 성격 등)
  - 운영/신규/프로젝트 등 업무 타입별 평균 포인트 분포
