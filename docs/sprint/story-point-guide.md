# 스토리 포인트 가이드 및 운영 방안

> YouTrack KB: DEV2-A-892 | 상위: DEV2-A-785 (Sprints)

## 1. 스토리 포인트(Story Point)의 정의

스토리 포인트는 작업에 소요되는 '시간'이 아니라, **작업의 상대적인 크기(Size)**를 나타냅니다.

- **판단 기준:** 복잡도(Complexity), 불확실성(Uncertainty), 작업량(Effort)
- **핵심 철학:** "이 작업이 며칠 걸리는가?"가 아니라 **"이 작업이 우리 팀에게 얼마나 큰 작업인가?"**를 정의하는 것

### 스토리 포인트를 도입하는 이유

1. **개인차 극복:** 숙련도에 따라 걸리는 시간은 다르지만, 작업 자체의 객관적 복잡도는 동일
2. **팀 속도(Velocity) 파악:** 우리 팀이 한 스프린트에 처리 가능한 평균 포인트를 파악하여 무리한 일정을 방지
3. **인식 동기화:** 산정 과정에서 팀장과 담당자 간의 요구사항 이해 차이를 사전에 발견
4. **현실적 계획:** 리스크와 불확실성을 수치에 포함하여 버퍼 없는 촉박한 계획을 예방

## 2. 산정 요소

산정 시 아래 세 가지 요소 중 하나라도 높다면 포인트를 상향 조정하여 판단합니다.

| 요소 | 설명 |
|------|------|
| **복잡도** | 비즈니스 로직이 얼마나 꼬여있는가? 예외 케이스가 많은가? |
| **불확실성** | 요구사항이 명확한가? 처음 시도해보는 기술이나 정책인가? |
| **작업량** | 단순 반복이라도 영향 범위가 넓거나 물리적인 작업량이 많은가? |

### 서비스 성격에 따른 직군별 역할 차이

| 구분 | 디자이너 역할 | 개발자(FE) 역할 |
|------|-------------|----------------|
| React 웹 서비스 | 피그마(Figma) UI/UX 디자인 | 디자인 가이드 기반 컴포넌트 및 로직 구현 |
| **레거시 서비스** | 디자인 + **퍼블리싱(HTML/CSS)** | 디자인/퍼블리싱 파일 기반 로직 연결 및 운영 |

## 3. 직군별 상세 산정 기준표 (1 ~ 13)

### 1) 개발자

구현 난이도, 데이터 변화, 기존 코드에 미치는 영향(Risk)을 종합 고려합니다.

| 포인트 (사이즈) | 로직 및 기능 구현 (Logic) | 데이터 및 인터페이스 (Data/API) | 영향도 및 리스크 (Risk) |
|----------------|--------------------------|-------------------------------|----------------------|
| **1 (XS)** | 단순 텍스트/컬러 변경, 상수 수정 | 기 구축된 환경변수 수정 | 영향 없음, 즉시 배포 가능 |
| **2 (S)** | 단순 분기 추가, 유사 기능 복사 수준 | API 단순 필드 추가, 단순 조회 | 영향 범위 국소적, 테스트 용이 |
| **3 (M)** | **[표준]** 신규 로직, 표준 CRUD 개발 | 신규 API 생성, 단순 테이블 설계 | 연관 모듈 회귀 테스트 필요 |
| **5 (L)** | 복잡한 예외 처리, 다수 상태 관리 | 복잡한 쿼리, 다수 테이블 조인 | 공통 모듈 수정, 높은 결합도 |
| **8 (XL)** | **[분할 권장]** 대규모 리팩토링 | DB 마이그레이션, 대용량 처리 | 장애 시 서비스 중단 리스크 |
| **13 (XXL)** | **[분할 필수]** 아키텍처 변경 | 인프라 교체, 다수 레거시 연동 | 한 스프린트 내 완료 불가 |

### 2) 기획자

정책의 확정성과 타 부서/기능과의 의존성을 중심으로 산정합니다.

| 포인트 (사이즈) | 기획 및 정책 수립 (Policy) | 비고 |
|----------------|--------------------------|------|
| **1 (XS)** | 문구 수정, 명확한 정책 한 줄 추가 | 즉시 결정 가능 |
| **2 (S)** | 기존 기능의 마이너 업데이트 정책 | 내부 검토 후 확정 |
| **3 (M)** | **[표준]** 신규 기능의 상세 UX 설계 | 타 부서 협의 필요 정책 |
| **5 (L)** | 복합적인 비즈니스 로직 설계 | 타 팀과의 데이터 연동 기획 |
| **8 (XL)** | **[분할 권장]** 서비스 도메인의 핵심 개편 | 법적/보안 리스크 검토 필요 |
| **13 (XXL)** | **[분할 필수]** 신규 서비스(모듈) 전체 기획 | 여러 단계의 기획/배포 필요 |

### 3) 디자이너

디자인 시스템 활용도와 결과물의 시각적 복잡도를 기준으로 산정합니다.

| 포인트 (사이즈) | React 웹 서비스 (Figma Only) | 레거시 서비스 (Figma + 퍼블리싱) |
|----------------|----------------------------|-------------------------------|
| **1 (XS)** | 아이콘 교체, 이미지 수정 | 간단한 HTML 태그 및 CSS 속성 수정 |
| **2 (S)** | 기존 컴포넌트 활용 페이지 구성 | 기존 레이아웃 내 정보 추가 및 퍼블리싱 |
| **3 (M)** | **[표준]** 신규 화면/컴포넌트 디자인 | 새로운 페이지 디자인 및 마크업 작업 |
| **5 (L)** | 복잡한 인터랙션 설계 | 레거시 레이아웃 구조 변경 및 퍼블리싱 |
| **8 (XL)** | **[분할 권장]** 섹션 전체 리뉴얼 | 복잡한 폼/인터랙션 포함 마크업 전체 |
| **13 (XXL)** | **[분할 필수]** 대규모 신규 과제 | 레거시 전체 CSS 테마/구조 개편 |

## 4. 협의 및 운영 규칙

### 팀장-담당자 협의 프로세스

1. **담당자 선제시:** 담당자는 위 기준표를 근거로 예상 사이즈(XS~XXL)를 먼저 제시
2. **검토:** 과거 유사 작업 데이터와 비교하여 의견을 나눔
3. **격차 해소:** 의견 차이가 클 경우, '불확실성'이나 '영향도'를 공유하여 합의점에 도달

### 13점(XXL) 관리 규칙

- **금지 사항:** 13점으로 산정된 티켓은 원칙적으로 스프린트에 그대로 담을 수 없음
- **분할 의무:** 반드시 8점(XL) 이하의 하위 작업들로 쪼개어 다시 산정

### 운영 가이드

- **포인트 = 작업량:** 숙련자가 10분 만에 끝낼 일이라도 객관적 난이도가 높다면 높은 포인트를 부여
- **블로커 발생:** 진행 중 예상치 못한 기술적/정책적 장애물 발견 시 포인트를 재협의
- **회고 반영:** 스프린트 종료 후, 실제 소요 시간과 포인트 간의 괴리가 컸던 작업들을 복기하여 기준표를 지속적으로 업데이트
