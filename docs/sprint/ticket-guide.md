# 스프린트 티켓 작성 및 운영 가이드

> YouTrack KB: DEV2-A-818 | 상위: DEV2-A-785 (Sprints)

## 1. 스프린트 상태 (상태값 기준)

아래 3개 상태는 스프린트 운영상 필수로 이행되어야 합니다. 그 외 상태는 담당자 재량에 따라 자율적으로 운영하되, 업무 종료 시점에는 반드시 "종료" 상태로 전환되어야 합니다.

- **백로그(Backlog):** 이번 스프린트에 진행하지 않을 "계획 업무" 티켓 상태 (필요는 있으나 이번 달 범위에서 제외된 항목 포함)
- **오픈(Open):** 해당 월 스프린트에서 진행할 "계획 업무" 티켓 상태. **스프린트 시작 시 계획 업무는 모두 Open 상태여야 함**
- **진행(In Progress):** Feature 및 관련 Task 작업에 착수한 상태. **동시에 task가 3개 이상 진행 상태로 있지 않도록 유의**

## 2. 티켓 단위 권장 범위

기본적으로 Epic은 월 단위 스프린트에서는 사용하지 않습니다. 다만 업무 범위가 월을 넘기거나, 여러 Feature를 묶어 관리할 필요가 있는 경우에는 사전 논의 후 Epic 사용으로 전환할 수 있습니다.

- **Feature:** 총 기간 1주일 이내 (필수). 1주일을 초과하는 Feature는 분할 필수
- **Task (Feature 하위):** 1일 이내 (필수). 1일을 초과할 것으로 보이면 Task를 더 쪼개어 작성

## 3. Feature 작성 필수 규칙: 5W1H 기반 작성 (필수)

**Feature는 반드시 5W1H 항목을 포함해 작성**합니다.

### What (무엇을 할 것인가)

이번 작업에서 바꾸거나 만드는 대상과 범위를 명확히 적습니다.
- 개발: 기능/모듈/API/배치/정책 로직 변경 범위
- 기획: 사용자 시나리오, 정책(허용/제한/예외), 화면/플로우 변경 범위
- 디자인: 적용 화면 범위, 컴포넌트 변경/추가, 상태(로딩/오류/빈화면) 정의 범위

### Why (왜 필요한가)

이 작업이 필요한 배경과 해결하려는 문제, 기대 효과를 적습니다.
- 개발: 중복 제거, 안정성/성능, 확장성, 유지보수성 개선 근거
- 기획: 운영 효율, CS 감소, 전환율/완료율 개선, 정책 준수 필요성
- 디자인: 가독성/일관성, 사용자 혼란 감소, 접근성/사용성 개선 근거

### Who (누가 사용할 것인가)

'사람 이름'이 아니라 **"사용 주체/대상"**을 적습니다.
- 사용자: 고객/관리자/파트너/내부 운영자 등
- 기획/디자인: 영향을 받는 이해관계자(운영팀/CS/파트너사 등)
- 개발: 호출 주체(어댑터/서비스/배치), 연관 시스템/모듈

### When (언제 수행/발생하는가)

'배포일/배치 시간'이 아니라 **"언제 이 기능이 동작하거나 이슈가 발생하는지(트리거)"**를 적습니다.
- 예: 로그인 시, 주문/결제 시, 상품 등록/수정 시, 정산 시, 특정 운영 액션 시, 스케줄 실행 시

### Where (어디에 적용되는가)

**"적용 위치"**를 적습니다.
- 기획/디자인: 적용 화면/메뉴/서비스 영역 (예: 마이페이지 > 구독, 어드민 > 상품관리)
- 개발: 적용 모듈/패키지/서비스

### How (어떻게 구현할 것인가)

구현 방식뿐 아니라, 정책/화면/검증 포인트까지 "핵심만" bullet로 적습니다.
- 개발: 주요 설계(인터페이스 분리, 트랜잭션/동기화, 예외 처리, 로그/모니터링)
- 기획: 정책 결정(허용/제한/예외), 운영 플로우, 데이터 기준(정의/소스)
- 디자인: 화면 구성/컴포넌트, 상태 정의(로딩/오류/빈 상태), 가이드/카피 반영

## 4. 팀원 표기 규칙

티켓 본문에서 팀원을 언급할 때는 **영어 이니셜(KJM, JEH 등)이 아닌 한글 이름**으로 표기합니다.

| 표기 | 예시 |
|------|------|
| ✅ 올바름 | 김정민, 조은흠, 안혜련 |
| ❌ 잘못됨 | KJM, JEH, AHR |

> 이니셜은 OKR 등 팀 내부 약어로만 사용하고, YouTrack 티켓에서는 누구나 읽을 수 있도록 이름을 사용합니다.

## 5. 예상 시작 일자 (필수)

모든 스프린트 대상 티켓은 작성 시 "예상 시작 일자"를 반드시 기재합니다. 시작 일자는 실제 착수일이 변동될 수 있으나, 계획 수립의 기준점으로 사용합니다. 작업 예측 시간과는 별개 요소입니다.

## 6. 스프린트 계획 수립 시점 (필수 운영 원칙)

모든 스프린트 대상 티켓은 **전월 마지막 주에 작성 및 계획을 완료**합니다.
(예: 3월 스프린트 티켓은 2월 마지막 주까지 작성/정리 완료)

## 7. 운영 원칙: 점진적 강화

티켓 관리 요소(필수 항목, 규칙, 체크리스트 등)는 한 번에 완벽히 고정하지 않고, 운영하면서 점진적으로 추가/강화합니다.

## 8. 이월 규칙 (운영 업무/계획 업무 공통)

스프린트 이월 시 반드시 코멘트로 사유를 남깁니다. 상세 절차는 [업무 계획 변경 절차](./plan-change-process.md)를 참조합니다.

## 9. 태그 컨벤션

모든 스프린트 대상 티켓(Feature, Task)에 아래 태그를 부여합니다.

| 태그 | 형식 | 예시 | 설명 |
|------|------|------|------|
| 역할 | `role:{역할}` | `role:개발`, `role:기획`, `role:디자인` | 담당 직군 |
| 서비스 | `biz:{서비스}` | `biz:만권당`, `biz:바자르`, `biz:투비` | 대상 서비스 |
| 스프린트 | `{YYMM}-planned` | `2604-planned` | 계획 수립 시점의 스프린트 월 |

## 10. YouTrack 필수 커스텀 필드

티켓 생성 시 아래 필드를 반드시 설정합니다.

| 필드 | 필수 | 설명 |
|------|------|------|
| Type | ✅ | Feature 또는 Task |
| State | ✅ | 스프린트 대상은 Open |
| Assignee | ✅ | 담당자 (서비스 owner 기본) |
| Service | ✅ | 대상 서비스 (만권당, 투비 등) |
| Sprints | ✅ | 스프린트 월 (2026.04 등) |
| Story points | Task만 | SP 산정 (Feature는 하위 Task 합산) |
| Platform | 선택 | Server, Web, iOS, Android 등 |
| Priority | ✅ | 기본 Normal |

## 11. OKR → 스프린트 변환 프로세스

분기 OKR을 월간 스프린트 티켓으로 변환하는 절차입니다.

```
분기 OKR (개인)
├── 해당 월 KR 항목 추출
│   └── 월별 포커스 테이블 참조
├── Feature 작성 (5W1H 필수)
│   └── 1주일 이내 완료 가능한 단위
└── Task 분해 (Feature 하위)
    └── 1일 이내 완료 가능한 단위, SP 산정
```

### 설계·분석 업무 SP 기준

코딩이 아닌 설계·분석·문서 작업의 SP 산정 기준입니다.

| SP | 사이즈 | 설계·분석 업무 기준 |
|----|--------|-------------------|
| 1 (XS) | 즉시 완료 | 미팅, 문서 정리, 모니터링 확인, 단순 배포 실행 |
| 2 (S) | 반나절~1일 | 단일 도메인 분석·설계, 설정 작업, 기능 검증, 패턴 조사, 전환/연동을 위한 영향도 분석·보조자료 작성 |
| 3 (M) | 1~2일 | 신규 아키텍처 설계, 복합 도메인 설계, 다수 이해관계자 협의 |

> 설계 업무는 코딩 대비 작업량은 적으나 불확실성·복잡도가 높을 수 있으므로, 의사결정 난이도를 반영하여 산정합니다.

## 12. YouTrack 마크다운 서식 규칙

YouTrack 티켓의 description은 마크다운을 지원한다. AI 도구(Claude Code 등)로 티켓을 생성/수정할 때 아래 규칙을 준수한다.

### 줄바꿈

MCP 등 API를 통해 description을 전달할 때 **실제 개행문자**를 사용한다. `\n` 이스케이프 시퀀스를 쓰면 리터럴 `\n`이 그대로 노출되어 한 줄로 붙는다.

| 방법 | 결과 |
|------|------|
| ✅ description 파라미터에 여러 줄 텍스트 직접 작성 | 마크다운 정상 렌더링 |
| ❌ `"## What\n내용\n## Why\n내용"` | 한 줄로 표시됨 |

### 지원 문법

헤딩(`##`), 테이블, 리스트(`-`), 인용(`>`), 코드블록(`` ``` ``), 굵게(`**`), 링크 등 표준 마크다운 문법을 그대로 사용한다.

## 13. 전사 YouTrack 상태 플로우

DEV2팀 스프린트 상태(위 1항)는 전사 YouTrack 상태 플로우 위에서 운영됩니다. 전체 상태 머신(ToBe → Submitted → Open → In Progress → Fixed → Closed 등)은 [YouTrack 티켓 가이드](../../youtrack/ticket-guide.md)를 참조합니다.
