# 스프린트 계획 운영 개요

## 월간 맨데이 배분 구조

한 달의 전체 맨데이를 아래 3가지 업무 유형으로 나누어 운영합니다.

```
전체 맨데이 (1개월)
├── 운영 업무 (비교적 단순 업무)
│   └── 버그 수정, CS 대응, 모니터링, 환경설정, 단순 문의 등
└── 계획 업무
    ├── 개발센터 과업 (OKR 연계)
    │   └── OKR KR에 직결되는 기능 개발, 아키텍처, 인프라 등
    └── 사업부 요청 (기능 요청, 급건 등)
        └── 사업부 기능 요청, 긴급 대응, 제휴 연동 등
```

### 배분 프로세스

1. **비율 사전 합의:** 월 시작 전, 운영 업무 : 계획 업무 비율을 팀장-담당자 간 협의하여 결정
2. **계획 업무 티켓 사전 생성:** 확정된 계획 업무 맨데이만큼의 티켓을 미리 작성 (5W1H 필수)
3. **스토리 포인트 산정:** 각 티켓에 스토리 포인트를 부여하여 총량이 맨데이에 맞는지 검증
4. **스프린트 태그 부여:** 해당 월 스프린트 태그를 붙여서 진행 대상 확정
5. **스프린트 시작:** 계획 업무 티켓은 모두 Open 상태로 시작

### Velocity 기반 계획 검증

월간 계획 업무 총량은 전월 Velocity와 해당 월 가용률로 검증합니다.

```text
계획 가능 SP = 전월 완료 Velocity × 0.8 × 가용률 보정
```

- 80% 계획 계수는 긴급 대응과 추정 오차를 위한 기본 버퍼입니다.
- OKR 직결 티켓을 먼저 배치하고, 잔여 수용량에 개선·고도화 티켓을 배치합니다.
- 월말에는 계획 이행률 → OKR 직결 완료율 → KR 달성률 순서로 스프린트 성과를 판정합니다.

### 운영 업무 vs 계획 업무

| 구분 | 운영 업무 | 계획 업무 |
|------|-----------|-----------|
| 성격 | 반응적, 단순, 반복적 | 선제적, 목표 지향적 |
| 예측 가능성 | 낮음 (돌발 발생) | 높음 (사전 계획) |
| 티켓 관리 | 발생 시 생성 | 월 시작 전 사전 생성 |
| 스토리 포인트 | 선택 | 필수 |
| OKR 연계 | 간접 | 직접 |

### 월간 운영 타임라인

| 시점 | 활동 |
|------|------|
| 전월 마지막 주 | 운영:계획 비율 합의 → 계획 업무 티켓 작성 → SP 산정 → 스프린트 태그 부여 |
| 월 1주차 | 스프린트 시작, 계획 업무 Open 상태 확인 |
| 월 중 | 긴급 요청 시 [업무 계획 변경 절차](./plan-change-process.md) 따름 |
| 월 마지막 주 (D-5 ~ D-day) | [스프린트 마감 프로세스](./sprint-closing-process.md) 진행 — 데이터 정합성, Velocity 산출, 다음 달 계획 확정 |
